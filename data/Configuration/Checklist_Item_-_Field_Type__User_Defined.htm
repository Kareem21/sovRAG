Checklist Item - Field Type: User Defined

The User Defined field type allows for filtering of options for users to select through the definition of the data source. This means that instead of the whole list, this can be reduced to only those that apply.

The data source can be from any Viewpoint database table or view so it is possible to filter lookup table values, user table values, drop-down values, Master Files, and even relationship types or share classes based on predefined option or user defined data source.

Note: Using user defined tags in Viewpoint Engage Workflow could potentially disclose private data to third party.

 

User Defined List based on Predefined Data Source

These are options that you select as the Data Source and can be:

  • Master File-specific

    • #PM.ST - Services for Time

    • #PM.SD - Services for Disbursements

    • #PM.MT - Matter Account

    • #PM.RF – Recurring Fee

    • #CA.NA - Nominal Account

    • #CA.SL - Sub-Ledger

    • #CA.SL.CC - Sub-Ledger based on Currency

  • Non Master File-specific (Global)

    • #CF.IR - Identity/AEOI Register Types

    • #CF.IRC - Identity Register Types

    • #CF.IRR - AEOI Register Types

    • #CF.RT - Relationship Types

    • #CF.AM - Account Movement Codes

    • #CF.OCT - Owners & Controllers Types

    • #CF.BIP - Binder Index Pages

You can also customise the code to further filter by type or code by specifying the data source like below:

<OptionCode>:<Type>

where:

  • <OptionCode> refers to the code of the selected option.

  • <Type> refers to the code of the type.

Option Code

Explanation and Example

PM.MT

This is to filter the Matter Account code.

 

Example:

PM.MT.G* - the list will show only matter code starts with 'G'.

CA.NA

This is to filter the Nominal Account based on the type or code.

 

Example:

CA.NA:F,2 - the list will show only Nominal Accounts where the account type is 'F' or '2'.

CA.NA.12*,20* - the list will show only Nominal Accounts where the account code starts with '12' or '20'.

CA.NA.14*,92*:F,4 - the list will show Nominal Accounts where the account code starts with '14' or '92' and the account type is 'F' or '4'.

Note: The types are from Nominal Accounts Setup.

CA.SL

This is to filter the Sub-ledger for the Nominal Account.

 

Example:

CA.SL.23*,4250 - the list will show only sub-ledgers for Nominal Account '4250' or Nominal Account starts with '23'.

CA.SL:B,C - the list will show only sub-ledgers where the Nominal Account type is 'B' or 'C'.

CF.IR

This is to filter the Identity Register Types.

 

Example:

CF.IR:F* - the list will show only Identity Register types start with 'F'.

CF.RT

This is to filter the Relationship Types. This also supports Relation Types based on the tag value configuration in Master File Type Lookup Table and Entity Type.

 

Example:

CF.RT:O - the list will show only Relationship types where the class is 'O'.

CF.RT.D*,S* - the list will show only Relationship types where the code starts with 'D' or 'S'.

CF.RT.A*,L*:E - the list will show only Relationship types where the code starts with 'A' or 'L' and the class is 'E'.

CF.AM

This is to filter the Account Movement Codes.

Example:
CF.AM:B,W - the list will show only Account Movement Codes where account type is 'B' or 'W'.

 

For all Master File-based option codes, filtering is allowed for other Master Files defined in the Master File field type with the following:

[FileCode=<KEY>]

where

  • <KEY> refers to the item key of the Master File field type where the value will be retrieved for

Note: When the <KEY> is not specified, the result will return an empty list.

For #PM.ST, #PM.SD and #CA.SL.CC, filtering can be done with the following:

[SourceInput=<KEY>]

where

  • <KEY> refers to the item key of the field where the value will be retrieved based on the condition. This can be lookup table, user table and drop-down.

User Defined List based on User-Defined Data Source

When setting up the checklist item, do the following:

  • Ensure the field type is set to User Defined

  • In Values/Default column, enter a name for the data source within Data Source field

  • Define the Data Source using tags in the workflow step properties - refer to Checklist/Workflow Step - Tags

In the workflow step properties, define the data source.

 

Examples:

[/OfficerList=|0:OfficerType|1:OfficerText|2:Officers|3:Status='A' AND RelClass IN('O','T') AND EntCode='(-MF-)'|/]

- This data source will display the code and description of active officer types (relationship types in Relation Class 'O' and 'T') of the Master File the workflow is for.

 

[/Shares=|0:ShareClass|1:ShareClassName|2:Share_Capital|3:NrShare<>0 AND EntCode='(-MF-)'|/]

- This data source will display the code and share class name of share classes that have not been cancelled for the Master File of the workflow.

 

[/BankList=|0:BankAddrCode|1:Name|2:BankAccounts AS A INNER JOIN RefMaster AS B ON A.BankAddrCode=B.RefCode|3:DateClosed IS NULL AND EntCode='(-MF-)'|/]

- This data source will display the code and name of banks where the Master File the workflow is for has an active bank account.

 

[/UtilityOptions=|0:ElCode|1:Name|2:UserTables|3:TabCode='ADVA' AND LEFT(Name,1)='U'|/]

- This data source will display only utility options from the ADVA user table.
