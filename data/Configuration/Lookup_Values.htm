Lookup Values

Throughout the system you will encounter lists which are fields containing predefined lookup values for selection. These lists are useful time-saving devices that also prevent data inconsistency.

This screen contains a list of the lookup tables that you can customise, if required. This includes:

  • adding tags to the lookup table

  • adding new values so that you have additional options from the list for the user to select

Lookup Tables Listing

Lookup tables that can be customised are displayed here. If the lookup table is not found here, this means that you are not able to customise the lookup table. The list shows both the code and name of the lookup table and you can sort it by clicking on the header of the column. To easily look for a lookup table, you can use the Find function by pressing CTRL + F keys. To select the lookup table, click on it.

  • Edit - Click to edit the selected lookup table in terms of:

    • Tag Values field - Tags will apply only to the selected Lookup Table.

      In the Tag Values field, you can enter:

      • Control whether new values can be added to the Lookup Table

        [ALLOWNEW=N] prevents users from adding new values to the Lookup Table manually by disabling the New button for the Lookup Value.

        For import actions either through Import/Export Settings or Data Replication, this tag is to be entered in the environment where the Lookup Table will be exported (Source/Master) environment.

        In Data Replication, the Master environment is used to set up the configuration, hence the [ALLOWNEW=N] tag does not limit the Master environment to create new Lookup Value.

        The [ALLOWNEW=N] tag and configurations from Master environment will then replicated to the Subcribers where the tag will take effect and disallow user in creating new Lookup Value.

        In the environment where import is performed (Destination/Subscriber):

        • when the exported .xml file contains value but the values do not exist in the destination/subscriber environment, these will be added.

        • when the exported .xml file does not contain value but the values exist in the destination/subscriber environment, these will be deactivated in the destination/subscriber environment.

        • when the exported .xml file contains the Lookup Table but the Lookup Table does not exist in the destination/subscriber environment, the Lookup Table will be imported.

        • when the exported .xml file does not contain the Lookup Table but the Lookup Table exists in the destination/subscriber environment, the Lookup Table will be deactivated in the destination/subscriber environment.

         If this tag is not entered, the default value is [ALLOWNEW=Y].

         

      • Specify the subscriber(s) of the Lookup Table

        • [SUBSCRIBER=<Subscriber>] is to specify the subscriber group(s) that the Lookup Table is for. For multiple groups, use a comma to separate e.g. [SUBSCRIBER=GEN,ASA,EUR]

        • [SUBSCRIBER<><Subscriber>] is to specify subscribers that will be excluded.

        See also:

        • Master Configuration Environment (MCE) and Data Replication

    • Risk Rating option - Tick this to apply risk scores for the values of the Lookup Table.

    • Customised option - Tick this to ensure that customisations for the selected Lookup Table will not be overwritten when an upgrade is performed.

    Note: The table code and name fields are displayed as reference. These fields are not editable.

 

Filter

The filter applies to the lookup table values for the selected lookup table where you can choose to view Active, Inactive or All values.

 

Lookup Values

This shows the values for the selected lookup table based on the selected filter.

 

Command buttons

Click on the relevant command button to perform the necessary.

  • New - Click this to add a new lookup value. If this is not active, this means that new values cannot be added to the lookup table.

    1. Enter details for the new value.

    2. Click Update.

     

  • Edit - Click this to edit an existing lookup value.

  • Global Change - Click this to update lookup values throughout Viewpoint. The global change screen that appears contain the following:

    1. In the global change screen, specify the change you want to make.

    2. Click Global Update.

    • Show All/With new code/Without new code - The Show All radio button is selected by default when you activate the Global Change screen, so all existing codes in the selected lookup table will be displayed. Select With new code or Without new code to isolate code with/without entries in the New Code column of the table.

    • Suggest - This function is available only for the Country Codes [CNTR] and Currency Codes [CURR] lookup value tables. When this button is clicked, the system will compare non-ISO standard codes to ISO-compliant ones and suggest the appropriate ISO-compliant code for the non-ISO standard one. If the ISO-compliant code suggested by the system already exists, the Delete check box will be ticked and a "[code] already exists" message will be displayed in the Message column.

      To allow the comparison, the ISO-compliant value will contain the tag [VPO=<OldCode>] where <OldCode> is the non-ISO standard code.

    • Current Code - This column displays the current codes in the lookup value table. If you click a cell in this column, a search button will be activated. Click this button to see where the selected code is being used.

    • Current Name - This column displays the current names that exist in the lookup value table. In some instances, such as in the Country Codes and Currency Codes tables, there may be duplicates or variations of a country name or currency.

    • VP Std. - This checkbox will be ticked to indicate they are Viewpoint's standard codes which cannot be deleted.

    • New Code - This column is where you enter the new code for an existing code. Doing so will add the [CVC=] tag with the old code to the Tag Values field.

    • Deactivate - Tick this to deactivate a code. Codes that are in use cannot be deactivated.

    • Delete - Tick this to remove the code.

    • Message - This will display system messages.

      Message

      Explanation

      Unable to delete/deactivate. Code is in use.

      This error may occur if you attempt to delete or deactivate a code that is being used.

      • If you are attempting to deactivate a code that is being used, we recommend that you do this in the Customize Lookup Values screen.

      • If you are attempting to delete a code that is being used, we recommend that you rename the code AND tick the Delete check box. This will rename the code and delete the old one.

      TABLE: <Table> FIELD: <Field>->The changes you requested to the table were not successful because they would create duplicate values in the index, primary key, or relationship. Change the data in the field or fields that contain duplicated data, remove the index, or redefine the index to permit duplicate entries and try again.

      This type of error may occur if you attempt to change a code to an existing code.

      • For example, if you change Guernsey [GG] to Guernsey [GI] in the Country Codes [CNTR] table. Since [GI] is already being used by Gibraltar, other areas such as the EntityType table may be affected: Gibraltar may have a Limited Company [LC] set up, and converting GG to GI may affect Gibraltar’s Limited Company [LC]. When this happens the change is aborted.

      • If you choose to ignore the warning, you can redo the change and tick the Ignore error during global update check box. If you do this, the change will be done on database rows that do not cause the error. All rows that generate errors will be skipped.

      [CODE1] already exists. Circular reference CODE -> CODE1 -> CODE2.

      [CODE2] already exists. Please update this code first to prevent circular reference.

      These messages appear if you are attempting to change one code to another code that already exists.

      • To overcome circular reference errors, rename CODE2 first, then CODE1;

      • or, use different codes for [CODE1] and [CODE2].

      [CODE] already exists.

      This normally happens when a code is being re-coded to a code that already exists.

      • Use a different code for the new code.

      [CODE] is used more than once.

      This normally happens when two or more codes are being recoded to the same code, as shown in the image above.

      • Use different codes for each of the new codes.

       

    • Global Update - Click to start the global update. Note that all other users should be logged out of Viewpoint before you do this. Results of the change will be logged in the LookValuesDecode_<TABLECODE>xx.csv file.

    • Ignore error during global update - Tick this to ignore warnings or errors that occur during a global update.

    • Close - Click to close the screen.
