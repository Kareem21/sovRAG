Relationship Types

Relationship types are used to identify the connection between Master Files. These types are divided into four classes that are used by Master Files, Entities and Trusts. Each class has its own relationship roles and a Master File can be tagged so that it is only to be used in certain relationship roles. Note that the system will, by default, not validate the usage of a Master File unless you enable it first.

A Master File and Entity relation can also have a corresponding or reciprocal relation, so that as one relation is created by the user, Viewpoint will create the other one. With reciprocal relations, any action performed on one will affect the other relation as well.

 

Filters

The filters apply to the relationship type list:

  • Type status - You can choose to view only active types, inactive ones or all.

  • Customised setting - You can choose to view types that have been customised or not.

 

Select/Apply button

This allows for additional criteria to apply to the relationship  type list.

 

Relationship Type list

This list shows existing relationship types that have been set up. By default, the relationship types are grouped by the class.

 

Command buttons

Click on the relevant command button to perform the necessary.

  • New - Click this to create a new relationship type.

    1. Enter details for the relationship type such as the code, class and description.

    2. Specify if the relationship type will be by:

      • Master File - This requires you to select the option for Principal Relation and enter the caption so that a Master File can be selected as the relation.

        • Any Master File - the selections available for this option are any Master Files allowed to be appointed as an officer or relation.

        • Appointed Relation - the selections available for this option are Master Files that are relations of the Master File where you are appointing the relationship type. This option is useful for the relationship type 'Alternate Director'.

        • Individual - the selections available for this are any Master Files allowed to be appointed as an officer or relation of the Master File Type 'Individual'.

        • Non-Individual - the selections available for this option are any Master Files allowed to be appointed as an officer or relation NOT of the Master File Type 'Individual'.

        • Not Used - this option indicates that this field is not used.

      • User - This requires you to tick User Relation option and enter the caption. This will allow a user account to be selected as the relation.

    3. Select additional fields for the relationship type such as:

      • Master File fields - Up to three Master File fields are possible and each can have its own caption. Set the relevant option for Additional Relation 1, 2 and 3 and specify the caption.

      • From Date fields - Up to three date fields are possible. Tick to select and specify the caption. When selected, these will be seen when you click Appoint.

      • To Date fields - Up to three date fields are possible. Tick to select and specify the caption. When selected, these will be seen when you click Resign.

      • User Table fields - Two fields are available to select values from User Tables. Tick to select and specify the User Table code.

      • Re-election - Tick this to allow re-election for the relationship type.

      • Position Text - Tick this to include a text field.

      • Describe Powers - Tick this to include a Key Powers field.

      • Reason Resignation - Tick this to allow Reason for Resignation field appear when you perform a resignation.

    4. If required, you can enter tags in the Tag Values field.

    5. Click Update.

    Note: To assign score for the types, tick the Risk Rating checkbox.

  • Edit - Click this to edit an existing relationship type such as changing its relation class or deactivating the relationship type.

  • Delete - Click this to remove an existing relationship type. If the relation is a Viewpoint standard, deactivate the relation by un-ticking the Active checkbox instead of deleting it as the next upgrade will re-insert the relation again.
