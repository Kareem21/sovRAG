Checklist Item - Field Type: Input Field

This field type is used to show or allow edit only relevant checklist items to the users based on:

  • selected type/jurisdiction/reporting authority/number of shares - will show or allow edit fields included for the selected type/jurisdiction/reporting authority/number of shares and applies to screens where the fields for the records differ based on the type, jurisdiction, reporting authority or number of shares such as:

    • Identity Register screen – user to select the type to see the relevant fields

    • Classification and Indicia screen – user to select the type to see the relevant fields

    • Reportable Accounts screen – user to select the reporting authority to see the relevant fields

    • Owners & Controllers screen – user to select the jurisdiction to see the relevant fields

    • All Relations and Officers screens – user to select the type to see the relevant fields

    • Interests screen – user to select the type to see the relevant fields

    • Capital screen – users are allowed to edit relevant fields based on the entity’s ownership structure

    • Shareholders screen – users to enter the number of shares to see the relevant fields

  • user's authority rights - will show fields that user have access and this applies to the screens above as well as screens where the fields do not depend on the type/jurisdiction/reporting authority and currently includes

    • Master File's General, Identity Details, CDD and Alerts screens

    • Entity File's General, M&A/Dates and Accounts screen

    • Billing File's General screen

    • Account File's General screen

The use of this field type requires items to be grouped with the parent item's field type set to Input Field. If this is for screens with different fields for its records, then there must be a sub-item for the selection of the type, jurisdiction or reporting authority that will determine what other sub-items will be displayed. Otherwise, the sub-items are just fields within the screen. The sub-items cannot be set as Input Field or have the Duplicate checkbox ticked.

Configuration - Parent Item

Apart from setting the field type for the parent item as Input Field, you must also specify the following in Values/Default:

  • Screen/Type - select the screen e.g. Identity Register [FMMCOMPR]

  • Screen Entry - select the action that will take e.g. &New [IDE.NEWR] to add a new identity Register record

Optional configurations are:

  • Duplicate - tick this to allow the group to be duplicated

  • Overwrite Caption - tick this to allow the item's caption to be overwritten by caption defined for the mapped field

 

Configuration - Sub-Items

The sub-items must be mapped to the field in the screen. When adding the sub-items, these can be done:

  • manually

    1. Click the icon to add item and indent it to make it a sub-item.

    2. Map the item to the field by:

      • Clicking the Browse button and specifying the Database Field in Clipboard Field; or

      • Assign the Database Field as the key for the item.

  • using Viewpoint Clipboard

    1. Configure a parent item as an ‘Input Field’ field type and define the Screen/Type and Screen Entry in the Values/Default column.

    2. Click the Paste - Viewpoint Clipboard icon. This will paste the fields as sub-items for the workflow step with the mapping in place.

If the workflow step is for screens with child grid such as Reportable Accounts and Owners & Controllers, then for this group, there must also be a sub-item for the selection of the reporting authority/jurisdiction and the following tags can be added:

  • [InputRefresh=1] - this is to refresh the sub-items for the child grid based to refresh the other sub-items.



  • [SourceInput=<Key>] - the <Key> refers to the key of the sub-item containing the reporting authority/jurisdiction value and the tag is added to filter values for the action type according to the reporting authority/jurisdiction..

Note: Filter tags, referencing tags and tags to mask the field description added to Lookup Values and User Tables will be applied to the workflow as well. 

 

Sub-Item with User Defined Field Type

When users defined a filter for the parent item with field type that is set to Input Field, the filter will be applied to all of the sub-items that are grouped under the parent item. If a filter is also defined in the sub-item, the values will be filtered based on the filter of the sub-item followed by the filter of the parent item. If the parent item is not configured with a filter, the values of the sub-item will be filtered based on the configuration of the Master File or Entity Type. Refer to Checklist Item - Field Type: User Defined for more information.

 

Example - Parent item is not configured with tag values

Values retrieved will be based on the data source of the sub-item and filtered by the Master File Type configuration of the Master File that executed the workflow.

The Master File Type configuration is referred to the tag values configured in the Master File Types [ADTP] lookup table in Configuration > Business Parameters > General > Lookup Values.

The Entity Type configuration is referred to the tag values configured in Configuration > Business Parameters > Entity Administration > Entity Types.

Note: If tag values are configured for both, Master File Types configuration will be prioritised, followed by Entity Types configuration.

 

Example - Parent item is configured with a tag value

The [/InputField.MFPROP=] tag is entered in the Tag Values field in the Edit Step Properties screen to filter the values shown.

Values retrieved in the sub-item will be filtered by Company Master File Type. The [FILTER:MAS.TYPE=<RelationshipTypes>] tag configured in Company Master File Type will be applied to all sub-items.

The data source of the sub-item is configured with a filter to show only relationship types that belongs to Class M. The values retrieved will be filtered by Class M and Company Master File Type.

 

Retrieve Balance Number of Shares

Following the support of Input Field for the Shareholders screen, the [+SYS:SHAREBALANCE.<Key>+] tag can be used to retrieve the balance number of shares of the selected share sequence number and be included in a workflow step. <Key> refers to the key in the workflow step that defines the share sequence number. Example: [+SYS:SHAREBALANCE.ShareSeqNr+]

 

Configuration for Partial Transfer, Redeem, Cancellation or Forfeit

Users are required to configure a conditional group for the items that will only be shown and allow for editing if the number of shares to transfer, redeem, cancel and forfeit are partial of the total amount of shares. After pasting the sub-items that mapped to the Transfer, Redeem, Cancel Cert. or Forfeit screens, users need to configure as follows:

  1. Insert a new checklist item below the item with the ‘CertificateNr’ key

  2. Define the key for the new checklist item as ‘Partial’

  3. Set the field type to Formula

  4. In the Values/Default column, enter the formula, V=IF(NrShare.V<>ShareTotal.V,1,0)

  5. Insert another checklist item below the Formula item

  6. Set the field type to Conditional Group.

  7. In the Values/Default column, enter Partial=1.

  8. Group all of the items below the Conditional Group item as the sub-item of the Conditional Group item.

  9. Click Update when completed.
