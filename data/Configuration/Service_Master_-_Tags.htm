Service Master - Tags

For the Service Master, there are tags that:

  • Apply to the service classes

    • Ad Valorem Fee [RV]

      When you set up a service as an Ad Valorem fee, you need to add the following tags. Each service will have its own tags:

      Tag

      Single Pricing Model

      Multiple Pricing Model

      [VALUE=X]

       

      This determines where the value is to be calculated from and X can be:

      • CANAV - Accounting NAV Valuation (default).

      • EAOI:<Reference> - Open Items from Entity Administrator for Open Item with the specified 'Reference'.

      Multiple sets of Ad Valorem structures on a single Ad Valorem is supported in the tag. This would cater for different pricing models while also combining different asset classes. It can be done by defining the initial tag ‘M0x’, where x starts from 1.

      The value includes Open Items, Bank Accounts, Securities and Fixed Assets from Entity Administrator.

      Asset Items

      Prefix

      Type

      Example tag

      Bank Accounts

      A

      BAType

      [M01.VALUE=EAASSET:A.BATYPE=CA]

      Explanation: This tag will retrieve bank accounts with the type ‘CA’ for ad valorem calculation.

      Note: Bank Account Type is retrieved from Lookup Values - Bank Account Types [BATP].

      Securities

      S

      SECType

      [M01.VALUE=EAASSET:S.SECTYPE=SL]

      Explanation: This tag will retrieve securities with the type ‘SL’ for ad valorem calculation.

      Note: Security Type is retrieved from Lookup Values - Security Types [SECT].

      Security Type can be seen in the Home > Accounting > Securities > Master screen - New/Edit - Security Type field.

      Open Items

      O

      OIType

      [M01.VALUE=EAASSET:O.OITYPE=LD]

      Explanation: This tag will retrieve open items with the type ‘LD’ for ad valorem calculation.

      Note: Open Item Type is retrieved from Lookup Values - Open Item Type [OITP].

      Fixed Assets

      F

      ASSETType

      [M01.VALUE=EAASSET:F.ASSETYPE=OT]

      Explanation: This tag will retrieve fixed assets with the type ‘OT’ for ad valorem calculation.

      Note: Fixed Asset Type is retrieved from Lookup Values - Fixed Asset Types [FATP].

       

       

      [CALC=X]

       

      This determines which NAV is used and X can be one of the following (except for [VALUE=EAOI:<Reference>] which will always use last NAV):

      • H - Highest NAV value in the period specified (default).

      • A - Average NAV value in the period specified.

      • L - Last NAV value in the period specified.

      Not supported in Multiple Pricing Model.

      [METHOD=X]

       

      Specify the method of calculation and X can be any of the following:

      • F - Flat percentage (default). This should be followed by [PERC=<Percentage>] e.g.

        [METHOD=F]
        [PERC=9]

      • S - Scaled percentage. This should be follows by [PSCALE=<AssetValue> ~<Percentage>] e.g.

        [METHOD=S]
        [PSCALE=
        1,000,000 ~ 1.5
        5,000,000 ~ 1.0
        999,000,000 ~ 0.5]

      • U - Percentage used for the NAV calculation to be specified using User Field. This should then be followed by [UFIELD] tags to define the User Field e.g.

        [METHOD=U]
        [UFIELD.Type=R]
        [UFIELD.Code=AVP1]

        [UFIELD.Type=R]

        The user field type used for the percentage (R = Matter User Field)

         

        [UFIELD.Code=<UserFieldKey>]

        The Key of the User Field.

         

        [UFIELD.Default=0]

        If a default value is specified, and the user field doesn't exist, the program will automatically create the field with the default value. Otherwise it will create the user field without any value.

      To perform validation on the percentage value of the User Field, you can use the following tags:

      [VALIDATE.MINPERC=<Value>] - User Field value must be bigger than the specified <Value> for this tag.

      [VALIDATE.MAXPERC=<Value>] - User Field value must be smaller than the specified <Value> for this tag.

       

      Supports all the methods in Single Pricing Model. For Multiple Pricing Model, the tag shall start with M0x, e.g.:

       

      [M01.METHOD=F]
      [PERC=9]

       

      [M01.METHOD=S]
      [PSCALE=
      1,000,000 ~ 1.5
      5,000,000 ~ 1.0
      999,000,000 ~ 0.5]

       

      [M0x.METHOD=U]
      [UFIELD.Type=R]
      [UFIELD.Code=AVP1]

       

      Additionally, Multiple Pricing Model also supports:

      • F- Flat percentage. This should be followed by [M0x.FSCALE=<AssetValue> ~<Percentage>] e.g. [M01.METHOD=F]
        [M01.FSCALE=
        10,000 ~ 0.2
        100,000 ~ 0.15
        999,000,000 ~ 0.1]

      Explanation:

      • If the total asset value < 10,000, it shall apply 0.2% of the total asset value.

      • If the total asset value > 10,000 and < 100,000, it shall apply 0.15% of the total asset value.

      • If the total asset value > 100,000 and < 999,000,000, it shall apply 0.1% of the total asset value.

      [FEE.MIN=<Amount>]

       

      To set a minimum fee amount that can be charged on annual basis.

      e.g.

      [FEE.MIN=100]

       

      For Multiple Pricing Model, the tag shall start with M0x.

      e.g.

      [M01.FEE.MIN=100]

       

      [FEE.MIN=U]

       

      To set a minimum fee amount based on the User Field. This is applicable for [METHOD=U] only. See [UFMIN] tags below:

       

      [FEE.MIN=U]

      [UFMIN.Type=R]

      [UFMIN.Code=<Code>]

      [UFMIN.Default=<Amount>]

       

      where:

      • U refers to the value defined in the User Field.

      • R refers to the User Field from Matter Account.

       

      To set a minimum fee amount based on the User Field. This is applicable for [METHOD=U] only. See [M0x.UFMIN] tags below:

       

      [M01.FEE.MIN=U]

      [M01.UFMIN.Type=R]

      [M01.UFMIN.Code=<Code>]

      [M01.UFMIN.Default=<Amount>]

       

      where:

      • U refers to the value defined in the User Field.

      • R refers to the User Field from Matter Account.

       

      [FEE.MAX=<Amount>]

       

      To set a maximum fee amount that can be charged on annual basis.

      e.g.

      [FEE.MAX=100]

       

      For Multiple Pricing Model, the tag shall start with M0x.

      e.g.

      [M01.FEE.MAX=100]

       

      [FEE.MAX=U]

       

      To set a maximum fee amount based on the User Field. This is applicable for [METHOD=U] only. See [UFMAX] tags below:

       

      [FEE.MAX=U]

      [UFMAX.Type=R]

      [UFMAX.Code=<Code>]

      [UFMAX.Default=<Amount>]

       

      where:

      • U refers to the value defined in the User Field.

      • R refers to the User Field from Matter Account.

      To set a maximum fee amount based on the User Field. This is applicable for [METHOD=U] only. See [M0x.UFMAX] tags below:

       

      [M01.FEE.MAX=U]

      [M01.UFMAX.Type=R]

      [M01.UFMAX.Code=<Code>]

      [M01.UFMAX.Default=<Amount>]

       

      where:

      • U refers to the value defined in the User Field.

      • R refers to the User Field from Matter Account.

      [M01.Fee.Min.<CurCode>=<Minimum Amount in CurCode>]

      Not supported in Single Pricing Model.

      This tag supports minimum amount based on the defined currency code.

      If there is no currency specified tag, the amount will be charged based on a generic minimum tag [M01.FEE.MIN=<Amount>]

       

      [M01.Fee.Max.<CurCode>=<Maximum Amount in CurCode>]

      Not supported in Single Pricing Model.

      This tag supports maximum amount based on the defined currency code.

      If there is no currency specified tag, the amount will be charged based on a generic maximum tag [M01.FEE.MAX=<Amount>]

       

      [AuMValue=A]

      Not supported in Single Pricing Model.

       

      This tag value allows user to specify the transactions retrieved for AuM calculation as well as to determine which exchange rate date to be taken for the calculation.

      For the exchange rate date, it is only applicable when the Matter currency is used. When the Reporting currency is used, the exchange rate will be based on the Transaction Date's rate.

      Note: M0x prefix is not required for this tag

      When the tag [AuMValue=A] is used, the system will first check if it is an existing client or new client.

      Note: If it is an existing client, the Recurring Fee’s Last Invoiced date shall have a value.

      When it is an existing client, the system will retrieve all the previous AuM transactions until the Recurring Fee’s ‘Fee Period From Date’ minus one (1) day.

      • Example: Recurring Fee’s Fee Period From Date = 1-Jan-2018. The AuM Transactions will retrieve AuM transactions until 31-Dec-2017. The foreign exchange date will also be 31-Dec-2017 if the Matter currency is used.

      When it is a new client, the system will retrieve all the previous AuM transactions until the last day of the month in the Recurring Fee’s ‘Fee Period From Date’.

      • Example 1: Recurring Fee’s Fee Period From Date = 15-Mar-2018. The AuM Transactions will retrieve AuM transactions until 31-Mar-2018.

      • Example 2: Recurring Fee’s Fee Period From Date = 1-Mar-2018. The AuM Transactions will retrieve AuM transactions until 31-Mar-2018.

      The foreign exchange date will also be 31-Mar-2018 for both the examples above if the Matter currency is used.

       

      [CHARGECUR=<Option>]

      Not supported in Single Pricing Model.

      This tag allows users to select the currency used for calculation.

      The currency option can be either Matter currency or Reporting currency. If there is no tag defined, the default currency will be Reporting currency.

      [CHARGECUR=<Option>]

      where <option> can be:

      • M = Matter currency

      • R = Reporting currency

      Note: M0x prefix is not required for this tag

      Example: [CHARGECUR=M]

      [CHARGETEXT=<Text to include/elaborate>]

      A default text can be used in the invoice line description. In order to do this, you need to add the round brackets ( ) to the Description field of the recurring fee. The default text will be placed between the brackets. The text can include the following fields that will be replaced with actual information on generation:

      • Asset Value: (-VALUE-)

      • Asset Currency: (-CCY-)

      • Annual Fee: (-FEEANNUAL-)

      • Period from: (-DATEFROM-)

      • Period to: (-DATETO-)

      • Fee charged (pro-rated): (-FEECHARGE-)

      The information for the field are as follows:

      • (-VALUE-) will retrieve the total value of the Open Item as at the Recurring Fee Due Date

      • (-FEEANNUAL-) will calculate the annual fee based on (-VALUE-). It will take into account the use of User Fields and tags specifying maximum or minimum fee amount.

      • (-DATEFROM-) will retrieve the (Recurring Fee Due Date - Number of Days of Recurring)

      • (-DATETO-) will retrieve the (Recurring Fee Due Date

      • (-FEECHARGE-) will be calculated as (Pro-rate) * (-FEEANNUAL-)

       

    • Minimum Time Fee [RM] and [RN]

      The time entries created for this service class will carry the recurring fee description, but you can customise it using the below:

       
      [MINFEECHARGE_DESCR=<Text for the minimum fee - charge entry>]

      Enter the above for the entry that is created for any shortfall.

      Note: This tag applies to both Service Classes RM and RN.

       

      [MINFEEREVERSE_DESCR=<Text for the minimum fee - carry forward of charge entry>]

      Enter the above for the reversal entry that is created for the carry forward entry.

      Note: This tag only applies to Service Class RM.

       

       

    • Transaction Fee [RT]

      Transaction Fee [RT] services are used when you want to charge a client for journals entered in Accounting, or for the documents stored on their behalf in Document Management.

      Journal Transactions

      The following are used when you want to charge a client for journals entered on their behalf.

      [TFTYPE=CATRANS]

      CATRANS identifies the transaction fee type, i.e. Accounting Transactions.

       

      [JournalCharge=<Option>]

      Specify how the charging is done:

      JOURNAL - Use this tag if you are charging the client based on journal entry.

      LINE - Use this tag if you are charging the client by individual journal lines.

      Eg: [JournalCharge=LINE]

       

      [ExcludeNominal=<Account>]

      Add this tag to exclude the named accounts from the journal charge calculation. If the journal charge type is by line, then the debit or credit of the line will also be excluded from the calculation. If no nominal accounts are excluded, then use NONE.

      Eg: [ExcludeNominal=1234|5678|1111], [ExcludeNominal=NONE]

       

      [UseDate=INPUT]

      Add this tag to calculate the journals based on the date of input. Leave this blank to calculate by journal date.

      Note: The Billing Event for this service should be set as Standard [STD].

       

      Document Management Storage

      The following are used when you want to charge the client for storing documents on their behalf:

      [TFTYPE=DMDOCS]

      DMDOCS identifies the transaction fee type, i.e., Document Management Documents.

       

      [DOCCOUNT=<Option>]

      Specify how the charging is done:

      FILES - Use this tag if you are charging the client based on the number of files stored.

      HOLDERS - Use this tag if you are charging the client based on the number of holders created.

      Eg: [DOCCOUNT=FILES]

       

      [INDEXFIELD=<IndexField>]

      Enter the Key Tag of the index field(s) that will be included when calculating the number of holders/files. Multiple index fields are separated by a pipe "|" symbol. The index fields entered must be of the field type Master File [40]. Use ALL for all index fields of the field type Master File [40].

      Eg: [INDEXFIELD=INDEX|CORRES|SIGNATORY]

       

      [DOCDATE=<Method>]

      Specify the calculation method:

      NEW - Use this to count only new files/holders in the date range. Note that the date referred to is the date the document was created, not the date the document is stored. This is to avoid charging the client multiple times for the same document.

      ALL - Use this to count all files/holders created before the Next Due Date of the fee.

      Eg: [DOCDATE=ALL]

       

      [CABINET=<CabinetCode>]

      Enter the Key Tag of the Cabinet(s) that are to be included in when calculating the number of holders/files. Multiple cabinets are separated by a pipe "|" symbol. Use ALL for all cabinets.

      Eg: [CABINET=CORGEN|LEGMINRES]

       

    • Time billed in advance with carry forward [TW] / Disbursement billed in advance with carry forward [DW]

      'Time billed in advance with carry forward [TW]' is used for invoicing the Fixed Fee in advance and later, to invoice in arrears any Time exceeding the Fixed Fee. It does the following:

      • invoicing Fixed Fee in advance.

      • track excess Time recorded against the Fixed Fee.

      • invoicing in arrears any Time exceeding the Fixed Fee.

      'Disbursement billed in advance with carry forward [DW]' is used for invoicing the Fixed Fee in advance and later, to invoice in arrears any Disbursement exceeding the Fixed Fee. It does the following:

      • invoicing Fixed Fee in advance.

      • track excess Disbursement recorded against the Fixed Fee.

      • invoicing in arrears any Disbursement exceeding the Fixed Fee.

      To enable the Service Class [TW] or [DW], the following tags need to be included:

      • [SRVDATE.Visible] is set to 'Y' to enable the ‘Service Date’ field in the Time/Disbursement Entry screen.

      • [SRVDATE.Mandatory] is set to 'Y' to set the ‘Service Date’ field as mandatory in the Time/Disbursement Entry screen.

      • [CHARGETIME_ACCT=<Account No>] is defined to bill the additional Time charged to the specified Account code. Similarly, [CHARGEDISB_ACCT=<Account No>] is defined to bill the additional Disbursements charged to the specified Account code. This is for posting to Client Accounting.

        Time/Disbursements billed in advance journal will be posted to the Client Accounting. Posting rule must be set up to ensure that source account feeds the correct release account. When Reconcile is run, the journal will get posted to the Release Account as defined in [CHARGETIME_ACCT=<Account No>] tag or [CHARGEDISB_ACCT=<Account No>] tag in the Service Master.

        Note: If the account defined is not active/not found, system will prompt a warning message that the account code defined in the tag is invalid.

      • [CHARGETIME_DESCR=<text description>] can be defined to provide the description of the additional Time charged in the invoice line. Similarly, [CHARGEDISB_DESCR=<text description>] can be defined to provide the description of the additional Disbursements charged in the invoice line.

      Note: When Service Class is 'TW' or 'DW', the tags [SRVDATE.Visible=Y] and [SRVDATE.Mandatory=Y] will be added by default. For other Service Class, [SRVDATE.Visible=N] will be added by default, meaning the 'Service Date' field will be disabled.

       

      User may proceed to invoice Time/Disbursements exceeding the fixed fee once completed the settings above and created the recurring fees under this service class.

      Note: Avoid using the same Service Code within the same Matter when creating Recurring Fees. If the same Service Code is used in the same Matter, the system will not be able to track the amount to be invoiced exceeding the fixed fee.

      To invoice Time/Disbursements exceeding the Fixed Fee,

      1. Go to the Time/Disbursements entry screen, create Time/Disbursements entry(ies) for Time/Disbursements exceeding the Fixed Fee. If the Fixed Fee has not been invoiced before, user will not be able to set the ‘Service Date’ for the Time/Disbursements exceeding the Fixed Fee. Besides, the 'Service Date' must be in the range of previously invoiced or current 'Fee Period From Date' and 'Fee Period To Date' of the Recurring Fee, otherwise a warning message will be prompted.

      2. After created the Time/Disbursements entries, proceed to post WIP on the additional Time/Disbursements and invoice them. Note: Time will be charged at Administrator rates based on the Rate Level set in File Maintenance > Sales Ledger > Billing File > General.

      Note: The Fixed Fee has to be invoiced first before the draft invoice for Time/Disbursements exceeding the Fixed Fee can be prepared.

       

      The invoiced Time/Disbursements exceeding the Fixed Fee can be seen in File Maintenance > Sales Ledger > Accounts Receivable > Receivables screen.

      Recurring Fees dates are automatically rolled forward to new dates when draft invoice of the Fixed Fee is invoiced.

  • Are used for calculation for Fees Payable such as corporate fee calculations, due dates and others

    [FEEDUE-MM=dd/MM]

    The fee due table determines the due date of the fee for payment to the registrar. Note the program will use the year of the recurring fees 'from date' to determine for which year the fee is payable.

    For example, if the incorporation month is January - June, and the fee is due on 31 May; and if the incorporation month is July - December, and the fee is due on 30 November, the tags would look like this:

    [FEEDUE-01=31/05]

    [FEEDUE-02=31/05]

    [FEEDUE-03=31/05]

    [FEEDUE-04=31/05]

    [FEEDUE-05=31/05]

    [FEEDUE-06=31/05]

    [FEEDUE-07=30/11]

    [FEEDUE-08=30/11]

    [FEEDUE-09=30/11]

    [FEEDUE-10=30/11]

    [FEEDUE-11=30/11]

    [FEEDUE-12=30/11]

     

    [PENALTY_MONTHS_x=y]

    There is a maximum of three penalty brackets, based on the number of months after the due date of the [FEEDUE]. x determines the penalty bracket and y determines the number of months after the due date of [FEEDUE].

    [PENALTY_MONTHS_1=2] - specifies that the first bracket is 2 months after the due date of the [FEEDUE].

    [PENALTY_MONTHS_2=5] - specifies that the second bracket is 5 months after the due date of the [FEEDUE].

    [PENALTY_MONTHS_3=0] - specifies that the third bracket is not used.

     

    [PENALTY_PERC_x=y]

    Use this tag to calculate the amount to be charged as penalty. x determines the penalty bracket and y is the percentage of the fee that will be charged. If you use this tag, do not use the [PENALTY_AMOUNT] tag. See also [FEETABLE] below.

    [PENALTY_PERC_1=10] - specifies that the first bracket penalty is 10% of the fee.

    [PENALTY_PERC_2=50] - specifies that the second bracket penalty is 50% of the fee.

    [PENALTY_PERC_3=0] - specifies that the third bracket penalty is not used.

     

    [PENALTY_AMOUNT_x=y]

    Use this tag to specify an absolute amount if you are not using the [PENALTY_PERC] tag above. x determines the penalty bracket and y is the absolute amount that will be charged. See also [FEECALC_CUR] below.

    For example:

    [PENALTY_AMOUNT_1=100] - specifies that the first bracket penalty is 100.00.

     

    [PENALTY_DESCR=x]

    Use this to define the description that will appear on the penalty invoice. x can be normal text and codes to retrieve information such as:

    • (-INVNR-) - refers to the invoice number

    • (-DRAFTNR-) - refers to the draft invoice number

    • (-DESCR-) - refers to the Service description

    • (-PERCENT-) - refers to the percentage of penalty being invoiced

    • (-PENALTYDATE-) – refers to the date the penalty being invoiced

    • (-STRIKEDATE-) – refers to the strike date

    For example:

    [PENALTY_DESCR=(-PERCENT-) Penalty on (-DESCR-) (Invoice# (-INVNR-)) (PENALTY_DATE (-PENALTYDATE-)) (STRIKEDATE (-STRIKEDATE-))]

     

    [STRIKE_MONTHS=5]

    This is the number of months after the [FEEDUE] date when the entity is struck off. Refer to Fees Payable - Strike Off .

     

    [FEECALC_BASIS=x]

    You can calculate the fee based on the following:

    [FEECALC_BASIS=AUTHCAP] - the fee is calculated based on the authorised capital and it will only be calculated when the authorisation type is AU.

    [FEECALC_BASIS=AUTHNUMBER] - the fee is calculated based on the authorised number of shares and it will only be calculated when the authorisation type is AU.

    [FEECALC_BASIS=AUTHCAPPREM] - the fee is calculated based on the total authorised capital plus the share premium.

     

    [FEECALC_METHOD=TABLE]

    This tag indicates to the Viewpoint that a scaled table is used for the fee calculation. See [FEETABLE] below.

     

    [FEECALC_CUR=USD]

    This tag specifies the currency for fee calculation. If the capital of the entity is in a currency other then this currency, Viewpoint will first look if there is an exchange rate entered with the share class and use this if there is. Otherwise, the capital is converted using the rate from the Exchange Rate Master.

     

    [FEETABLE]

    The following is an example of the fee table. The number left of the tilde (~) is the upper band of the fee scale, and the number right of the ~ the fee. The table can have as many lines as required. If the number of shares is 0 or if the entity has no authorised capital, then the lowest fee band will be used.

    [FEETABLE=

    50,000 ~ 450

    99999999 ~ 1200]

     

    [FEECALC_PIT_DATE_DD/MM]

    This tag is for Point-In-Time Date. This is the day/month for the Point-In-Time calculation.

    Note: This is also valid for existing fee calculation basis. If specified, it will do Point-In-Time, otherwise means it is not Point-In-Time.

    For example: [FEECALC_PIT_DATE=31/08], [FEECALC_PIT_DATE=01/02]

     

    [FEECALC_PIT_YEAR=XXYY]

    This tag is for Point-In-Time calculation. XX can be either one of the following:

    DF - Fee Period From Date of the Recurring Fee

    DT - Fee Period To Date of the Recurring Fee

    +Y or -Y is the number of years to add/subtract. The number of years must be around value and it should not be in 0.5 year.

    For example:

    Recurring Fee Period From Date = 01 Jan 2020

    Recurring Fee Period To Date = 31 Dec 2020

    [FEECALC_PIT_YEAR=DT-2], this means the year will be retrieved as 2018.

    [FEECALC_PIT_YEAR=DF+10], this means the year will be retrieved as 2030.

  • Will set applicable analysis code

    If you are using the Analysis Code feature, you can set certain analysis codes to be available for the service for users to select when entering time:

    [ANALCODES=<AnalysisCode1>,<AnalysisCode2>,...]

    Ensure that the codes have been set up in the Analyses Codes Time Entry [TANALYSES] user table, in Business Parameters > General > User Tables/Values. Wildcard is possible for this tag. For example, [ANALCODES=T01*] will allow all analysis codes starting with 'T01'.

     

    Note: If this tag is not entered for the service, all active analysis codes will be shown. The analysis code is case sensitive, please ensure it is entered in the correct case.

  • Will set default analysis code

    If you are using the Analysis Code feature, you can set default analysis code for the service:

    [DEFAULTANALCODE=<AnalysisCode>]

    Ensure that the codes have been set up in the Analyses Codes Time Entry [TANALYSES] user table, in Business Parameters > General > User Tables/Values.

  • Will enable Service Date field

    When using the 'Service Date' field with other Service Class (i.e. not TW/DW), this field will only function to record the date of the service in a Time/Disbursement entry. If you wish to enable this field, you can set the tags below:

    [SRVDATE.Visible=Y]

    This will enable the 'Service Date' field in Time/Disbursement Entry screen.

     

    [SRVDATE.Mandatory=Y]

    This will set the 'Service Date' field as mandatory in Time/Disbursement Entry screen.

     

    Note: By default, [SRVDATE.Visible=N] will be added for Service Class that is not [TW] or [DW], meaning the 'Service Date' field will be disabled. For Service Class [TW] or [DW], please refer to the section above - 'Time billed in advance with carry forward [TW] / Disbursement billed in advance with carry forward [DW]'.

  • Are used for batch maintenance of payable fees

    If you would like purchase orders to be generated for batch payment, you can use the following tags to define details for the purchase order as well as when the purchase order is to be generated.

     

    [PO_SUPPLIER=<SupplierFileCode>]

    This is to set the default supplier for the purchase order. The value set here will be seen in the Supplier field in the Government Fees screen.

     

    [PO_FEES=<ServiceCode>]

    This is to set the default fee or service for the purchase order. The value set here will be seen in the Supplier field in the Government Fees screen.

     

    [PO_PENALTY=<ServiceCode>]

    This is to set the default penalty fee or service for the purchase order. The value set here will be seen in the Supplier field in the Government Fees screen.

     

    [Max_Amount=<Amount>]

    This tag is used to set the maximum total amount of fees in a payment batch. A purchase order will be created only if the maximum total amount is not exceeded.

    Example [Max_Amount=12345]

     

    [Max_Lines=<Number>]

    This tag is used to set the maximum number of payment in an active payment batch. A purchase order will be created only if the maximum number of lines is not exceeded.

    Example [Max_Lines=100]

  • Excludes column(s) update during import from the master configuration

    [NOUPDATE=<DBField>,<DBField>]

    <DBField> refers to the database field in service master screen. Client can put multiple <DBField> by adding the ',' in between the database field.
