Key Information

 

The Key Information field of a template contains additional codes or queries that will determine the information retrieved for the template. Depending on the template type, the following can also be added to the Key Information:

  • Word Templates

    • Document Management index tags

    • Action Management tags

    • Special codes

  • HTML Templates

    • Grid Attributes

    • Context Menu Statements

 

Queries and Statements

For both Word and HTML templates, the queries and statements are used for information retrieval and can be divided into the following:

Item

Explanation

Header statement

This depends on the selection of the Select Definition field for the template but can be customised by the user. It is not applicable to Forms.

For HTML templates, this is used only with the ftyp=evw prefix.

 

Syntax:

[/HD-SQL=SELECT .../]

 

Example:

When Select Definition = Entity Information [015], the header statement would be

[/HD-SQL=SELECT * FROM CR_Entity WHERE/]

 

Billing templates

Billing templates have two header statements: one for proforma invoices and the other for invoices. Each template must have both, regardless if they are used only for draft, invoice, or both types of documents.

[/HD-SQL=SELECT * FROM ProformaHeader WHERE /]

[/HD-SQL!=SELECT * FROM InvoiceHeader WHERE /]

 

HTML templates

HTML templates must also have the following:

[/HD-SelKey=<Field>='(-MF-)'/] - to link the selected Master File to the Master File field of the header statement e.g.  [/HD-SelKey=RefCode='(-MF-)'/] means value for the RefCode field is the selected Master File

[/HD-SelRef=MF=<Field>/] - to define the Master File field to use for checking on whether the user can access the Master File or not e.g. [/HD-SelRef=MF='(-RefCode-)'/] means the RefCode field will determine if the user has access to the Master File

For both, <Field> is a field retrieved by the header statement e.g. (-EntCode-),(-RefCode-)

 

Note: Ensure that the query does not contain any '&' (ampersand), '[' ']' (square brackets) and '[/' '/]' (square brackets slash). These characters are the reserved query characters and cannot be used by the user.

SQL query

This is defined by the user to specify the database table or view required to obtain the necessary information.

For HTML templates, this can be used with both ftyp=evw and fgrd prefixes.

 

Syntax:

 [/<Query>=SELECT.../]

where <Query> is a unique name for the query.

 

Example:

[/CoType=SELECT ShortName FROM EntityTypes INNER JOIN Entity ON (EntityTypes.EType=Entity.EntType AND EntityTypes.JurisDic=Entity.IncorpPlace) WHERE Entity.EntCode='(-EntCode-)'/]

 

Note: Ensure that the query does not contain any '&' (ampersand), '[' ']' (square brackets) and '[/' '/]' (square brackets slash). These characters are the reserved query characters and cannot be used by the user.

DT query and statement

This is also defined by the user to obtain information which will be presented in rows and columns. The query is to specify the database table or view required to obtain the information and the statement is to define the formatting for the retrieved information by the corresponding query.

For HTML templates, this is used only with the ftyp=evw prefix.

 

Syntax:

For the query - [/DT.<Query>-SQL=SELECT.../]

For the statement- [/DT.<Query>-FORMAT=\<Column>:<Format>\.../]

where

  • <Query> for both the query and statement must be the same

  • <Column> refers to the column number

  • <Format> is the Format Option code

 

Example:

[/DT.BUSNAME-SQL=SELECT BusRegNr, PrincipleBNR, BusNames.DateRegistration, BAddrNr, BusName, DateRenew, BusNature FROM BusNames WHERE ((EntCode='(-ENTCODE-)'))/]

[/DT.BUSNAME-FORMAT=\2:Yes/No\3:DMB\4:#AC-1-1\6:DMB/]

 

Note: Ensure that the query does not contain any '&' (ampersand), '[' ']' (square brackets) and '[/' '/]' (square brackets slash). These characters are the reserved query characters and cannot be used by the user.

 

Header Statement - Field Codes

When your header statement selects all fields (*) from a table or view, you can retrieve any of the fields in the template using the prefix HD.

For example, with 

[/HD-SQL=SELECT * FROM CR_Company WHERE /]

you can use (+HD.Name,+), (+HD.EntClient+), etc.

 

Customising the Header Statement

The header statement can be changed such as replacing * with actual fields that you require, adding condition(s) or adding ORDER BY to sort the retrieved information.

To add condition(s), you need to add the condition and 'AND' to the header statement - [/HD-SQL=SELECT * FROM <Table/View> WHERE <Condition> AND/]

To sort information, add ORDER BY - [/HD-SQL=SELECT * FROM <Table/View> WHERE <Condition> AND ORDER BY <Field>/]

 

How do I get information for all entities into one report?

This is normally required for reports that provide a snapshot of information in the database at a particular time, such as entities that are due for renewal in the first half of the year. Use the COUNT function in the header statement for situations such as this.

Note: You may not be able to use the HD prefix for your field codes, and will have to retrieve the information using normal SQL queries.

 

When using COUNT, the conditions specified in the selection criteria screen will have no impact on the results, and you need to make sure that your conditions are specified in your SQL queries.

For example:

[/HD-SQL=SELECT COUNT(EntCode) FROM CR_Shareholders WHERE /]

 

I am getting multiple copies of the same report. What should I do?

Generally, the addition of the keyword 'DISTINCT' to the header statement will resolve this issue. You may need to further customise the query so that you get the exact results that you want.

For example:

[/HD-SQL=SELECT DISTINCT EntCode FROM CR_Shareholders WHERE /]

 

Same Selection Criteria as HD-SQL

Using (-HDSEL-) will get the exact conditions as HD-SQL query.

Word Report Statement Billing File - with Outstanding Invoices [RMSTBFE2]

[/OUTINV=SELECT O.RefCode, O.BillAddress, O.RevEntity, O.HeaderNote, O.DateCreate, O.EntClient, O.DateDue, O.DateInvoice, O.ProformaNr, O.InvoiceType, O.CurCode, O.InvoiceCur

FROM BR_OverDueAgeingCur O WHERE RefCode = '(-RefCode-)' AND RevEntity = '(-RevEntity-)' AND CurCode = '(-CurCode-)' AND (-HDSEL-) ORDER BY DateInvoice/]

 

XML Report

[/ADDRESSES=SELECT * FROM CR_RefAddresses WHERE Active = 1 AND (-HDSEL-)/]

 

Word Report Statement Billing File - Include Outstanding Invoices without Draft [RMSTBFINV1]

[/OUTINV=SELECT RefCode, ProformaNr FROM BR_OverDueAgeingCur WHERE RefCode = '(-RefCode-)' AND (-HDSEL-)/]

 

Index Tags

These refer to the Document Management index tags which can be added for Word templates in the following libraries:

  • Word Reports

  • Form Library

  • Correspondence

  • Billing Templates

The tags are obtained from Index Wizard and placed here.

 

Action Management Tags

These tags apply only to Billing templates and will automatically create a new action when the document is generated. This way, there is no need for user to manually create the action. Refer to Action Management for more information.

 

Special Codes

These are additional codes that can be added for specific purposes as follows:

Code

Usage

[/DELETE_FORM=<Code>/]

 

where <Code> refers to the code of the template

  • Used in the Form Library only.

  • Deletes the change links for specified forms to disable the forms when replacements are imported into the Form Library.

  • Normally used for statutory forms.

Example:

[/DELETE_FORM=FRM123,FRM145/]

The above will delete the change links for forms FRM123 and FRM145.

 

[/SAVETO=<FileName>/]

 

where <Filename> can be normal text or field code or both

  • Used in Word Reports, Correspondence, and Billing Templates libraries only.

  • Allows you to specify the file name to use when saving the generated document.

  • For field code, this must be fields retrieved by the header statement and must be defined using this syntax '(-Field-)'.

Example:

[/SAVETO=(-RefCode-) Draft Invoice (-ProformaNr-)/]

The above will result in a file name that contains the code of the Billing File (RefCode) and the draft invoice number (ProformaNr).

 

Note: For posted invoices, the above needs to be repeated with an exclamation mark '!' after SAVETO, as in:

[/SAVETO!=(-RefCode-) Invoice (-InvoiceNr-)/]

[/HD-Mailto=RefCode/]

 

  • Used in Word Reports and Billing Templates libraries only.

  • This will ensure the Master File column in the E-mail Bin picks up the name of the Master File when the generated document is sent to E-mail Bin.

[/PRINT_DIALOG=Y/]

  • Used in the Form Library only.

  • Prevents users from editing the form after generation.

  • Adding this code to the Key Information will display the Output Document screen when users click on Print. This allows users to choose whether to print the form or convert it to a PDF file.

 

[/EDIT_DIALOG=<UserGroup/]

where <UserGroup> refers to the code of the User Group.

  • Used in the Form Library only.

  • Prevents users in the specified User Groups from editing the form after generation.

  • Adding this code to the Key Information will display the Output Document screen when users click on Print. This allows users to choose whether to print the form or convert it to a PDF file.

Example:

[/EDIT_DIALOG=ADMIN,MGRS,SYSMGR/]

means that users in these three User Groups will see the Output Document screen, and are not able to edit the generated form before it is printed or converted to

PDF.

[/GROUPDOC=<Field>/]

  • Used to merge Word reports into a single Word file where the group <field> is the same.

  • The <field> must be a database field returned by the header statement and the header statement must be sorted by this field.

  • If [/GROUPDOC=*/] is used, you will only get one Word file. This is equivalent to the obsolete [TO_ONE_FILE] option.

  • When using this code, do not use Viewpoint (+...+) codes in the header or footer of the document template.

Example:

[/GROUPDOC=*/]

will merge all generated documents to one file.

 

[/HD-SQL=SELECT * FROM CR_Company WHERE ORDER BY EntClient, EntCode/]

[/SAVETO=Register of Directors -  Client Group Code: (-EntClient-)/]

[/GROUPDOC=EntClient/]

will result in one generated document with the filename containing the code of the client Master File e.g. 'Register of Directors - Client Group Code: ARCHER.doc'

[/LINKCODE.<ChangeCode>=<GridCode>/]

where

  • <ChangeCode> is the change code that triggers the form

  • <GridCode> is the code for the grid containing the record that you wish to link the form to

  • Used in the Form Library only.

  • Links generated form to a record in the specified grid once the form has been stored.

  • This code can be used multiple times in a form template to create record links in different grid for different changes.

  • The list of Change Codes are available in the Change Codes screen.

  • For the Grid Code, click here to learn how to obtain the code.

    1. Go to  the screen containing the grid.

    2. Click Grid Details tab on the right.

    3. The gridcode is located on the bottom of the screen.

    Alternatively, right-click within the grid and click View Details.

[/MERGE_EVENT=<ChangeCode>/]

where <ChangeCode> can be:

  • OFP for Officer/Relation - Change of Particulars

  • SMP for Shareholders - Change of Particulars

  • BEP for Beneficiary - Change of Particulars

  • ITP for Entity Interest Holder - Change of Particulars

  • ONC for Owners & Controllers - Change of Particulars

  • Used in the Form Library only.

  • Used to report multiple changes to one Master File as one change.

  • This is normally used for change of particulars forms.

Note: This is required if the form includes [/GOSUB=OFFGENERAL/] sub-routine, see below.

 

[/GOSUB=<Sub-Routine>/]

  • Used in the Form Library only.

  • These codes are used when the form requires a specific sub-routine in order to properly retrieve information.

  • There can only be one sub-routine per form.

For example:

[/GOSUB=OFFGENERAL/]

will retrieve:

  • details of appointed officers where the officer code starts with D,S or M

  • details of change of particulars for Master Files that are Statutory or Trust Officers, Entity Relations, Shareholders, Beneficiaries, Interest Holders, Owners and Controllers.

This is normally used for change of particulars forms where the BLOCK statement within the template should contain the condition EGRP=OFFGEN.

 

[/GOSUB=ALLT/]

will retrieve information related to issuance of shares in a specific format. Examples of field codes for this sub-routine are:

(+SUB.IssueAmount+) to get the currency and amount of shares issued

(+SUB.NewAllotC+) to get details of shares issued like the share class, currency, number of shares issued, amount paid per share and amount unpaid per share.

You can see this sub-routine in these templates for Hong Kong - HK14NSC1 and HK14NSC11.

 

[/GOSUB=ShareIssueNrs/]

allows for SQL queries in the template to refer to share issue number to retrieve related information.

For example, when this sub-routine is added, the CanCSh query contains the condition 'IssueNr IN (-ShareIssueNrs-)'

 

[/GOSUB=ShareIssueNrs/]

[/CancSh=SELECT SUM(NrShare*-1), ShareClassName, CASE WHEN Currency='GBP' THEN NCHAR(0xA3) WHEN Currency='EUR' THEN NCHAR(0x20AC) WHEN Currency='USD' THEN NCHAR(0x024) ELSE Currency END AS Currency, NomValShare FROM CR_ShareTransactions WHERE EntCode='(-EntCode-)' AND IssueNr IN (-ShareIssueNrs-) GROUP BY ShareClassName, Currency, NomValShare/]

You can see this sub-routine in this template - GI14NTCNCLSH.

 

[/GOSUB=CHANGEPRTCOFF/]

will retrieve details of change of particulars for statutory officers with hard-coded field codes and requires the use of BLOCK statements with certain conditions.

You can see this sub-routine in these templates for Gibraltar - GI14PTSEC, GI14PTDIR, GI14PTMGR.

 

[/GOSUB=CHANGEPRTCSHR/]

will retrieve details of change of particulars for shareholders with hard-coded field codes and requires the use of BLOCK statement with the condition EGRP=SHPX.

You can see this sub-routine in these templates for Gibraltar - GI14MEMDET.

 

Note: These sub-routines are also available but they apply only to certain jurisdictions and the information retrieved is very specific - OfficerGib, OfficerMal, OfficerLab, OfForGib, OfForIOM and OfFReignMa.

[/FRMHKG=<Sub-routine/]

  • Used in the Form Library only.

  • Similar to [/GOSUB=<Sub-routine>/].

  • These codes will retrieve certain information in a specific manner and is based on requirements and design of the template. If the design of the template changes, an enhancement is required.

  • There can only be one sub-routine per form.

  • Examples:

  • [/FRMHKG=CYHE57/]

  • [/FRMHKG=CYHE4/]

  • [/FRMHKG=D2B/]

[$MAIL]

This is automatically added when the template is refreshed if the template contains E-mail Tags.

[/FILEEXT=<Format>/]

This tag is used to allow .txt templates to be saved as another format such as .xml or .csv. The tag is useful for templates that are to be submitted to authorities who require it in a specific format such as .csv.

 

Note: A number of .txt templates generate as .xml file by default so this tag is not necessary.

[/Editor.Mode=<mode>/]

where <mode> should be specified as 'Comparison'

This tag is used for Data Comparison. It will enabled the XML Viewer which showing the 'Retrieve Data' ribbon button.

Example:

[/Editor.Mode=Comparison/]

[/Editor.<Column>=<ColumnCaption>/]

where <Column> is either 'CompareFrom' or 'CompareTo'

where <ColumnCaption> is the intended caption to put for specified column during Data Comparison

This tag is used to change the default column caption in the XML Viewer during Data Comparison to the caption as per defined in the template's Key Information.

Example:

[/Editor.CompareFrom=Viewpoint Data/]

[/Editor.CompareTo=Authority Data/]

[/ComparePatterns=<Element>,<Element>…./]

[/<Element>.Pattern=<Format>/]

where <Element> is to specify the element especially identifier e.g. DateOfBirth.

where <Format> is the format of the identifier to be formatted to e.g. yyyy-MM-dd

The value of the specified element in the tag will be formatted to the defined format if this tag is included in the Key Information of a template with the mode 'Comparison.' This will improve the accuracy of the Data Comparison.

Because the date format returned from Authority data could not be matched with the format for Viewpoint data, this is especially useful for elements with date formatted data

Example:

[/ComparePatterns=DateOfBirth,StartDate,EndDate,DateOfRegistration/]

[/DateOfBirth.Pattern=yyyy-MM-dd/]

[/StartDate.Pattern=yyyy-MM-dd/]

[/EndDate.Pattern=yyyy-MM-dd/]

[/DateofRegistration.Pattern=yyyy.MM.dd/]

[/Editor.CaseSensitive=Y/

 This tag is used in the template's Key Information during Data Comparison, and it allows each data to be case sensitive. The data will be in case insensitive format by default if this tag is not defined in the template's Key Information.

[/Editor.WhitespaceSensitive=Y/]

This tag is used in the template's Key Information during Data Comparison to allow for whitespace sensitive leading and trailing spaces on each data set. If this tag is not defined in the template's Key Information, the data will be insensitive to whitespace by default.

 

Tips on Coding

  1. Use single quote ( ' ) instead of double quotes ( " ) EXCEPT for abbreviations(datepart) for DATEADD, DATEPART, DATEDIFF functions in MS Access.

    • For MS Access: DateAdd("yyyy",1,AcctYearEnd).

    • For MSSQL: DateAdd(yyyy,1,AcctYearEnd).

  2. Use brackets around all keywords:

    • SELECT [Open],[Left].

    • SELECT DateAppoint AS [Date].

Grid Statements

These are for HTML templates where the prefix fgrd is used in the template to display retrieved information in a grid with additional features such as ability to sort by column and a Refresh button.

 

The grid statement is as follows:

[/Query>-<Setting>=<Value>/]

 

where

  • <Query> is the name of the grid statement and must be unique

  • <Setting> refers to the attribute of the grid

  • <Value> refers to the required information for the <Setting> and differs based on the <Setting>

 

<Setting>

Syntax

Explanation

SQL

[/<Name>-SQL=<Statement>/]

This is to define the query for retrieving information.

 

Example:

[/OEREL-SQL=SELECT MeetType, MeetName, MGTT.[Name], CASE WHEN MeetActive = 0 THEN 'Inactive' ELSE 'Active' END AS MeetActive, EntCode, MeetCode FROM EntMeetType EMT LEFT JOIN qLvFNRP MGTT ON MGTT.ElCode = EMT.MeetFreq WHERE EntCode = '(-EntCode-)'/]

The above specifies that the OEREL grid will be retrieving information from EntMeetType.

CAPTION

[/<Name>-CAPTION=\1:<Caption1>\2:<Caption2>\:.../]

 

where <Caption1>,<Caption2>,.. will be text that will appear as the column heading.

This is to define the column header for each column of the grid. Each column is separated by a '\:'.

 

Example:

[/OEREL-CAPTION=\1:Meeting Type\2:Meeting Name\3:Frequency\4:Status/]

The above specifies that the grid will have four columns and the headings are 'Meeting Type', 'Meeting Name', 'Frequency' and 'Status'.

WIDTH

[/<Query>-WIDTH=\1:<Width1>\2:<Width2>\:.../]

 

where <Width1>,<Width2>,..values will add up to 100.

This is to define the column width for each column of the grid. Each column is separated by a '\:'.

 

Example:

[/OEREL-WIDTH=\1:20\2:40\3:30\4:10/]

The above specifies that the first column will have a width of 20, the second column width is 40, the third column width if 30 and the fourth column width is 10.

FORMAT

[/<Query>-FORMAT=\1:<Format1>\2:<Format2>\:.../]

 

where <Format1>,<Format2>,... is the format which can be:

  • DMB – Date format

  • DATETIME – Date & time

  • C2 – 2 decimal points for amount

  • C4 – 4 decimal points for amount

  • C0 – No decimal points for amount

  • C6 – 6 decimal points for amount

  • PEC – Percentage

  • DTM – Time

  • NS – Not show

  • CM=<Id> – Context menu

  • Viewpoint Format Option codes

This is to define the format for values in the column. Each column is separated by a '\:'.

 

Example:

[/OEREL-FORMAT=\1:\2:\3:\4:\5:*NS\6:*NS/]

The above specifies format for the 5th and 6th column. For the other columns, no formatting is specified.

 

Note: The format must be in order so if there is no format required for a column, leave it empty.

CM

[/<Query>-CM=<Option1>,<Option2>,…/]

where <Option1>,<Option2>,... refers to the workflow that you would like to make available for selection. However, <Option1>, <Option2> is not the name of the workflow template but rather, the name of the Context Menu Statements used to define the workflow.

This is used to specify the workflow that can be started when you right-click on a Master File option in the grid. Each workflow is linked to the option which must be further specified using the Context Menu Statements.

 

Example:

[/OEREL-CM=APPREL, RESREL/]

The above specifies two options for the context menu and properties for the option will refer to additional context menu statements in the Key Information with the name 'APPREL' and 'RESREL'.

 

Context Menu Statements

These statements are required for context menu options that you want to add to the template.

For HTML template with the ftyp=evw prefix, apart from the View Details, Open File, Enquiry and Relations options, you can specify more options that will open other Viewpoint screen. To do this, add the context menu statement in the following syntax:

[/CM.<Identifier>=<Screen>/]

 

where

  • <Identifier> is the field with the context menu

  • <Screen> is the screenID

Example:

In Key Information - [/CM.Info=FSEGENER,FSEMNADT/]

In the template - (+ftyp=evw|CM.EntClient\#INFO+)(+ftyp=evw|HD.EntClient\#AD-1-0+) (+ftyp=evw|CM.EntClient\#INFO|END+)

 

For HTML template with the fgrd prefix, additional context menu options would be for starting workflows where both the caption and the workflow template for each option must be specified.

Syntax

Explanation

[/<Option>-CAPTION=<Caption>/]

This is used to specify the caption that will be seen when user performs a right-click to access the context menu.

 

Example:

[/APPREL-CAPTION=Appoint Relation/]

[/<Option>-URL=WF.mfCode=<Field>&Wkey=<WorkflowTemplateKey>/]

where

  • <Field> is to define the Master File for the worklow

  • <WorkflowTemplateKey> is the Key of the workflow template

This is used to launch a workflow from the context men and requires the workflow template key and the Master File for the workflow.

[/APPREL-URL=WF.mfCode=(-HD.EntCode-)&Wkey=VPWFR001/]

The above will launch the workflow with the key 'VPWFR001' for the Master File selected within the grid.

[/<Option>-URL=WF.mfCode=<Field>&Wkey=<WorkflowTemplateKey>&defaultValue=<Item>/]

where

  • <Field> is to define the Master File for the worklow

  • <WorkflowTemplateKey> is the Key of the workflow template

  • <Item> refers to a checklist item within the workflow and must be defined with the item key and the default value enclosed within square brackets:

    [ItemKey=<DefaultValue>] e.g. [HD.ACT=Appoint of Entity Relation]

Note: It is possible to specify multiple default values but each must have the item key and the default value.

This is similar to the above but with default values specified for items in the workflow.

Example:

[/APPREL-URL=WF.mfCode=(-HD.EntCode-)&Wkey=VPWFR001&defaultValue=[HD.ACT=Appointment of Entity Relation]/]

The above will launch the workflow with the key 'VPWFR001' for the Master File selected within the grid where the checklist item 'ACT' will have the default value 'Appointment of Entity Relation'.

[/APPREL-URL=WF.mfCode=(-HD.EntCode-)&Wkey=VPWFR001&defaultValue=[HD.ACT=Appointment of Entity Relation][HD.RTYPE=Registered Agent]/]

The above will launch the workflow with the key 'VPWFR001' for the Master File selected within the grid where the checklist item:

  • 'ACT' will have the default value 'Appointment of Entity Relation'

  • 'RTPE' will have the default value 'Registered Agent'

 

Buttons and Hyperlinks

When the page has button and hyperlink, the following must be specified for the button/hyperlink:

  • the caption

  • the workflow template

There are two types of button/hyperlink behaviour:

  • When clicked, it will launch the Workflow based on the record from the grid.

    [/<fgrd Name>-RD=<Function Name>/] 

    [/<Function Name >-URL=<URL value>/]

     

    In HTML page:

    For button:

    <input type="button" value="button caption" onclick="FunctionName()"/>

     

    For hyperlink:

    <a onclick="FunctionName()"> hyperlink caption</a>

     

    Example (for button):

    [/OEREL-RD=GetAddrCode/]

    [/GetAddrCode-URL=WF.mfCode=(-HD.EntCode-)&Wkey=VPWFR001&defaultValue=[HD.ACT=Appointment of Entity Relation(s)][HD.C1=Yes][HD.AddrCode=(-AddrCode-)]/]

     

    In HTML page: <input type="button" value="Appoint Relation" onclick="GetAddrCode()"/>

     

  • When clicked, it will launch the Workflow in standalone manner (not retrieving information from the grid).

    [/<Function Name>=<URL value>/]

     

    In HTML page:

    For button:

    <input type="button" value="button caption" onclick="(+ftyp=evw|WF.FunctionName+)”/>

     

    For hyperlink:

    <a onclick="(+ftyp=evw|WF.FunctionName+)">hyperlink caption</a>

     

    Example (for button):

    [/AppointRelation= WF.mfCode=(-HD.EntCode-)&Wkey=VPWFR001&defaultValue=[HD.ACT=Appointment of Entity Relation(s)][HD.C1=Yes]/]

     

    In HTML page: <input type="button" value="Appoint Relation" onclick="(+ftyp=evw|WF.AppointRelation+)"/>
