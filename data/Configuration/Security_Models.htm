Security Models

Viewpoint uses highly customisable role-based security settings which are defined by a combination of User Roles, User Group Roles, Administrator Groups and, where applicable, Access Groups. It is recommended that you understand how the different models work and select the most suitable one before you proceed to customise and use Viewpoint.

 

Security Elements

The following are the parts that make up Viewpoint's security setup.

 

User Roles

User Roles define the roles and authorities of a user. A User Role is similar to a job responsibility while the authority determines how high a level of responsibility a user has. You can create as many or as few User Roles as you need, and each role is then assigned to one or more User Groups.

 

User Groups

Users Groups are groups of users who are defined by the jobs they do and the responsibilities that they have. You need to assign at least one User Role to a User Group, or the users would have no access to Viewpoint. Depending on how you have set up your User Roles, you can assign just one role to a group, or you can assign multiple roles. If the roles that you assign to the group have overlapping authorities, then the highest of the authorities will be taken as the group's authority level. A user can only be assigned to one User Group, so you need to set up as many User Groups as required to cater for this.

 

Administrator Groups

Administrator Groups can be used to represent departments or, in cases where an organisation has offices in various jurisdictions, it can also represent jurisdictions. Master Files can be assigned to only one Administrator Group, but a user can be assigned to more than one group if he needs to be able to access the files of more than one group. However, the functions he can perform on the Master Files are determined by his User Group.

 

Access Groups and Access Group Security

Access Groups are used only if you implement the Access + User Group Security Model [3] as your security model. Access Groups can be combined with any number of Administrator Groups and User Groups so that a user who is assigned an Access Group will be able to access the Master Files in the associated Administrator Group and perform the functions that have been given to the associated User Group. This forms the Access Group Security.

The user's own Administrator Group and User Group are ignored for all Master File-related screens, such as the Statutory screens and the Sales Ledger screens, but are used for access to non-Master File related screens such as the Reporting screens.

 

Security Models

The six models are described below.

 

User Group Authority Only [1]

This is the default security model where you can only control access to the screens and functions in Viewpoint. This means that a user will be able to access all Master Files and perform the same functions for all files. To implement this model, do the following:

  • In Viewpoint Management, Database > Settings > Application, set Security Model = User Group authority only [1].

  • In Configuration > Users & Security, set up the following:

    • User Roles

    • User Groups

    • Administrator Groups - Although this does not affect Viewpoint security, you will still need to link this when setting up a user

    • Users - File Access option is to be set to 'Administrator Groups + View-only for Others [4]'

 

Administrator + User Group Model [2]

This model allows for the following control:

  • Access to the screens and functions in Viewpoint.

  • Access to Master Files.

This means that you can restrict a user from accessing certain Master Files so they will not be able to open the Master Files or view details of the file. However, if the user is allowed to access a Master File, then the screens and functions that they can perform depend on the user's User Group.

The restriction will be based on the Administrator Group and the File Access option set for the user account. To implement this model, do the following:

  • In Viewpoint Management, Database > Settings > Application, set Security Model = Administrator + User Group model [2].

  • In Configuration > Users & Security, set up the following:

    • User Roles

    • User Groups

    • Administrator Groups

    • Users - ensure that the right File Access option is selected

 

Access + User Group Model [3]

This model also allows for the following control:

  • Access to the screens and functions in Viewpoint.

  • Access to Master Files.

It is similar to the Administrator + User Group Model [2] except that access to the screens and functions for a user can differ based on the Master File. This means that the same user can have different access rights for different Master Files.

For this model, access to screens that are Master File-independent will depend on the user's User Group but access to Master File and screens for the Master File will depend on the Access Group Security which will take into account the Access Group of the user and the Administrator Group of the Master File.

To implement this model, do the following:

  • In Viewpoint Management, Database > Settings > Application, set Security Model = Access+ User Group model [3].

  • In Configuration > Users & Security, set up the following:

    • User Roles

    • User Groups

    • Administrator Groups

    • Access Groups

    • Access Group Security

    • Users - ensure that the right File Access option is selected

Note: For this model, although the user may have authority to the Master File, the Access Group Security will determine whether or not the user will be able to open the Master File.

 

User Relation Links [4]

This model works in the same way as the Administrator + User Group Model [2] except that the system will check on User Relations for the Master File to control access to the Master File. To implement this model, do the following:

  • In Viewpoint Management, Database > Settings > Application, set Security Model = User Relation Links [4].

  • In Configuration > Users & Security, set up the following:

    • User Roles

    • User Groups

    • Administrator Groups

    • Users - ensure that the right File Access option is selected

  • In Configuration > Business Parameters > Entity Administration > Relationship Types, set up the relevant User Relations.

For the Master File, ensure that the required User Relations are added.

 

Services Provided Model [5]

This model allows for two types of control:

  • Restriction of Master Files to users - Users can access a Master File i.e. open the Master File if the user belongs to the same Administrator Group as the Master File.

  • Different authorities for different Master Files - Users can have different authority rights for the different Master Files that the user can access via the Services Provided screen.

  • Restriction to view Administrator Groups - Users can view other Administrator Groups if the user belongs to the same Administrator Group.

  • Restriction to view other Administrator Groups Users - Users can view other Administrator Group users if the user belongs to the same Administrator Group.

 

Regardless of the control type, you can also choose to prevent a user from being able to see a Master File that the user cannot access. This means that if the user does not have access to a Master File, the user will not be able to see the Master File at all e.g. performing a Search Master File using the Master File name will not turn up any result.

To implement this model, do the following:

  • In Viewpoint Management, Database > Settings > Application, set Security Model = Services Provided Model [5].

  • In Configuration > Settings > Tools & Utilities > Localise, run the Enable Services Provided Security Model to:

    • Set the Services Provided screen to active.

  • In Configuration > Users & Security, set up:

    • User Roles (and Services Provided if you want to have different authorities for different Master Files)

    • User Groups

    • Administrator Groups

    • Users - ensure that the right File Access option is selected

      Access Level

      Access to Master File

      0 - No Files

      No access to any Master File.

      2 - Administrator Groups

      The user has authority and can only access Master Files that are in the same Administrator Group.

      The user will NOT be able to see Master Files in another Administrator Group.

      The authority to edit is to be based on the Service Provided.

      4 - Administrator Groups + View-only for Others

      User can access to the Master Files that do not belong to their Administrator Group, but the authority is limited to View-only based on the User Groups authority.

       

For the Master File, the following must be done to ensure access rights are granted:

  • Set the Administrator Group in the Setup screen.

  • Add the relevant services provided to the Master File in the Services Provided screen for each Administrator Group.

Note: A record will be added automatically to the Services Provided screen when the Administrator Group is assigned in Setup. The record created will have a default service where the default service is retrieved from the 'Primary Office Default Service' field in Configuration > Settings > System Default > Security screen. The similar applies when changing the Administrator Group for the Master File.

 

Service Office Model [8]

This model allows for access to a Master File and the screens within the Master File to be controlled based on the services provided to the Master File from a service office. This means that a user:

  • May not be able to access a Master File if the Master File does not have any services provided by the service office of the user.

  • Can access a Master File but see only screens based on the services provided to the Master File by the user's service office.

However, even if the user does not have access to a Master File, if the user performs a search for the Master File, the user will be able to see the Master File if it exists.

This model takes into account the following concepts:

  • Service Office - This is the Administrator Group and in this model, a Master File can belong to more than one Administrator Group. The Administrator Group in the Master File's Setup screen is known as the primary service office. Other service offices assigned to the Master File in the Service Office screen are the secondary service offices.

  • Master File Owner - This refers to users who belong to the same Administrator Group as the Master File's primary service office. This can be the user's default Administrator Group or any other Administrator Group of the user.

  • Services Provided - This refers to User Roles that are tagged with 'SP' in the Electronic Filing Code field in the User Roles [URLS] lookup table so that it appears in the Service Office screen for selection.

To implement this model, you need to do the following:

  • In Viewpoint Management, Database > Settings > Application, set Security Model = Services Office Model [8].

  • Set up the following in Configuration > Users & Security:

    • User Roles and Services Provided

    • User Groups

    • Administrator Groups

    • Users - ensure that the right File Access option is selected

  • Set the appropriate Service Office Authorisation method for screens in Configuration > Settings > Customise > Windows Menu.

    Service Office Authorisation Method

    Then..

    User Group Role [0]

    Users can access the screen if their User Group has authority to access the screen.

    This is the default screen setting.

    Combined UGR+SP [1]

    User can access the screen if his/her User Group's Roles coincides with the service provided to his/her Service Office. If the user's Group roles do not provide him with access to the screen in his own service office, he will not be able to access the screen in the secondary service office even though the Services Provided allow for access to the screen.

    If the user's Group roles provide him access to the screen in his own service office, and the Services Provided allow for access to the screen in the secondary office, the lower of the access authorisation will be granted to the user.

    This screen authorisation level does not take into account the user's Administrator Group/Service Office.

    MF Owner+UGR [2]

    If the user belongs to the same Administrator Group/Service Office as the Master File, he will be given access to the screen per his/her User Group Roles.

    MF Owner+UGR else UGR+SP [3]

    If the user is of the same Administrator Group/Service Office as the Master File, his/her screen access will follow his User Group Roles (MF Owner+UGR [2]), otherwise, it will check for a combination of the Services Provided and user's Group roles (Combined UGR+SP [1]).

     

For the Master File, the following must be done to ensure access rights are granted:

  • Set the primary service office in the Setup screen.

  • Add the relevant services provided to the Master File in the Service Office screen.
