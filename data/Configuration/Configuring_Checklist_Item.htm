Configuring Checklist Item

What you require the items to do will determine the configuration of the items in terms of the field type to use and whether or not you need to enter information in the Values/Default column.

 

Configuring Item as Instructions or Headings

If the item is only to provide information to user, then you can set the field type as ‘Not Applicable’.

Example – Step to instruct user to provide information.

 

Configuring Item to Accept User Input

For item to accept and store user input, you must ensure that the right field type is set. The field type depends on the expected input value e.g. set field type to Date if you want user to enter a date. The field type MUST NOT be ‘Not Applicable’ if it is to accept input.

Example – Step to get information for a process.

Configuring Item to Retrieve Information

It may be required for the step to display information that is already in the database. This can be part of the item’s caption or item’s value and uses field codes. If part of the item's caption, then field code is to be entered in the Checklist item column as (-<Field>-); or (-<Query>.<Field>-) if SQL query is used.

If information is to be retrieved as value for the item, then ensure that the field type matches the information to be retrieved and the field code is entered in the Values/Default column.

Example – Step to get information from Accounts screen.

Note: For checklist items with the (+HD.<Key>+) code, duplicating them will remove the default value and you can then enter a new value.

 

If information is to be retrieved based on another checklist item that is completed by the user, ensure that a key is assigned to the item that is to be referred and enter the tag [+WF.<Field Key>\<Format>+] in the Values/Default column.

Example – Step to get information based on another checklist item that is completed by the user.

Note: Refer to Format Options for more information.

 

If the information required is not in the Workflow Field Wizard, use SQL query to retrieve the information and insert field code that refers to the SQL query.

Example – Step to get the Registered Agent of the company.

 

If there is more than one instance of the information to be retrieved, then you can add COPY tag so that you only define one item. This item will be repeated if more than one record is returned.

Example – Step to get the name of the active Directors for the company.

 

When there is more than one type of information, the above can also be used but you can define different items and group them.

Example – Step to get name, date of appointment and date of resignation of past Directors for the company.

Retrieve Information from PDF XFA Form

Information in a PDF XFA form can be retrieved and automatically populated into the workflow.

 

The configuration required is as follows:

Have a checklist item with ‘Document Holder’ field type to store the form. In the ‘Store Action’ field, enter ‘EXTRACTXML|REFRESH’.

  • 'EXTRACTXML’ is for extracting the content of form into the clipboard and store it in the database.
  • 'REFRESH’ will take effect after ‘EXTRACTXML’ process where it will refresh the workflow content.

The items which values are to be retrieved from the form must be mapped accordingly by entering the field key from the form into the ‘Clipboard Mapping’ field. If the checklist item in the workflow is having the same key as the item in the form, you can use ‘#’. Field type for the items is according to the detail to be retrieved.

Configuring Item to Retrieved Required Items

If the workflow contains required items that you would like user to complete in another workflow step, a checklist item with the field type ‘Required’ must be set up with the key of the workflow step containing the required items and the key of the required items.

 

Example – Workflow to set up a Billing File where the first step is to get details for the Billing File and the second step retrieves required items from the first step.

For the first step, all the required items must be assigned a key and have the Required checkbox ticked.

For the second step, to have the checklist item retrieve the required items from the first step, the field type must be set to ‘Required’ and refer to the required items.

 

Configuring Item to Retrieve Assignee Information

Details of assignee entered in Viewpoint such as telephone number and e-mail can be retrieved for the workflow by adding these prefixes to the field code of the detail to be retrieved:

  • DF.USERWF – If the assignee is the assignee of the workflow.

  • DF.USERWS – If the assignee is the assignee of the workflow step.

You can refer to the Field Wizard in Document Library for field codes of user details that can be retrieved.

Note: For assignee of the workflow step, this must be the same workflow step i.e. you cannot retrieve details of assignee for previous workflow step.

 

Example – Step to get name and telephone number of the assigned user of the workflow step.

 

Configuring Item to Retrieve Activities Related Information

Details of activities and their linked records can be retrieved for the workflow by adding these prefixes to the field code of the detail to be retrieved:

  • AT – If the information is regarding the activity record.

  • AL – If the information is regarding the linked record of the selected activity record.

Format options are applicable to the field codes to retrieve the information in a specific format.

Example – Retrieve Master File of the activity record in company name format and the name of the user who set the alert that is linked to the activity record.

 

Configuring Item for Viewpoint Clipboard

If the user input in the workflow is to be updated to a Viewpoint screen, then you can set it up so the values entered can be copied and pasted using Viewpoint Clipboard. This removes the need to re-enter the information. The configuration requires:

  • Items containing the information to have field type that follows the input value and mapped using the database field.

  • Item of the field type 'Go to Screen' or 'Master File (with New Master File option ticked)' to contain CLIP tags that will perform the copy and paste. Refer to Checklist Item - Tags for more information on CLIP tags.

Example – Workflow with step to update information to Accounts screen using Viewpoint Clipboard to copy and paste information from the workflow.

 

Example - Workflow with step to add new appointment in Statutory Office screen with appointment details from workflow.

For this example, it is possible to have the item open the Statutory Officers with the appointment details automatically pasted. This requires the following:

  • Items containing the information to have field type that follows the input value and mapped using the database field.

  • Item of the field type 'Go to Screen' with CLIP tag added and Screen Entry specified.

 

If the information to be copied is from an earlier workflow step, then add the following:

(CLIP.Source=<StepKey>)

where <StepKey> refers to the key of the earlier workflow step. If the information is from multiple items in the earlier workflow step and the step is a repeatable step, then add {REF} to ensure the information copied is according to its group.

 

Example – Workflow for adding new bank signatories with a repeatable step for details of bank signatories and a step to show information from the earlier step and create Master File for new signatories with information copied and pasted using Viewpoint Clipboard.

Step for getting details of bank signatory to:

  • Have the option ‘Allow Copy Step’ ticked.

  • Contain items configured with the items mapped using the database field.

  • Field type for the items to depend on expected input value.

 

Step to create new Master File(s) to:

  • Contain items showing values from earlier step using CCODE and COPY tag.

  • Contain item with field type ‘Master File’ with CLIP tags to specify earlier workflow step and {REF}.  

 

Same Database Field

If the database field name is the same e.g. NrShare for number of shares for a share class and number of shares to be issued, add the Change Code as part of the key to differentiate them. The syntax will be as follows:

<ChangeCode>.<DBField>

For example, NrShare is the field name for number of shares for a share class and number of shares to be issued. As such, to differentiate them, the Key will be:

·         SHH.NrShare – refers to number of shares for a share class

·         SHA.NrShare – refers to number of shares for share issue

Note: You can get the Change Code from Viewpoint - Home > Configuration > Business Parameters > General > Change Codes.

 

Contact Information

For copying information to the Contact Details grid in Addresses screen, add the below to Clipboard Field:

$CONTACT.<ContactType>

where <ContactType> refers to the code of the contact information type.

Configuring Item as Screen Shortcut

If it is required for you to access screens from the workflow, you can have items to act as shortcut to the screen. The field type will be ‘Go to Screen’ and the screen will be specified in Values/Default column by selecting from the list of screens.

Apart from just going to the screen, it is also possible to apply an action in the screen. In a screen with grid, it is as if you have just clicked on a command button within the screen.

Note: List options that start with a ‘+’ is a screen heading (for example, to indicate report type) and not a valid screen. It serves as a guide for the screen selection.

 

Example - Process to check that information has been entered in CDD screen.

The field type will be ‘Go to Screen’ and the screen will be specified in Values/Default column by selecting the screen from the list.

When you click on the icon for the checklist item, this will open the screen in File Maintenance.

 

Example – Step to resign selected officer.

Checklist items for the step include items that will contain details for the resigning officer and a checklist item to go to the Statutory Officers screen.

For the details, these will have the relevant field type, and assigned a key as well as mapped to the relevant field. For the checklist item that will bring you to the screen, the field type is ‘Go to Screen’ and the following must also be defined in the Value/Default column:

  • Screen Entry – select the action to perform by selecting the relevant command button

  • Criteria – specify the details of the record for the action using the Database Field and pipe symbol ‘|’ to separate the fields

 

Note: The Criteria should contain details that will allow the program to identify the record correctly. If there is no existing record matching the Criteria, the action will not be performed. If there is no Criteria specified, then the action will apply to the first record in the screen.

 

Additional Tags

There are also tags that can be added for the screen shortcut.

 

[Embed=Y] Tag

If you would like the screen to be embedded in the workflow, add the tag [Embed=Y]:

With the tag, clicking on the icon will open the screen in the workflow:

 

You can work on the embedded screen normally i.e. enter and update information.

With the above, the screen would be for the same Master File as the workflow e.g. if the workflow is started for the Master File ‘Jones – Sally’, the screen will also be for ‘Jones – Sally’. If you want the screen to be for another Master File:

  • The other Master File must be selected in the workflow step e.g. there is an item with a Key containing the Master File

  • Add the tag [FileCode=<Key>] where <Key> refers to the key for the item containing the Master File

 

[CLIP.Set=Y] and [CLIP.Clear=Y] Tags

These tags can be added for Viewpoint  Clipboard’s copy function to be performed automatically once user clicks on the icon to go to the screen.

If you would like the information copied to also be automatically pasted without the user manually clicking on a command button e.g. Appoint, select the option '&Appoint[SOF.APPT]'  in Screen Entry. This means that once user click on the icon to go to the screen, it will be as if the user have also clicked on the Appoint command button.

If the information is to be copied and pasted to an existing record, then you can also define the record using the Criteria field. For this, define the record as follows:

<DBField>=(+WF.<Key>+)

where

  • <DBField> refer to the database field containing the value

  • <Key> refers to the Key of the item where the value is retrieved

Example:

AddrCode=(+WF.MF1+)

will look for the record where the AddrCode is the same as the Master File specified in the item with the Key ‘MF1’.

If required, you can have multiple criteria by separating each with a ‘|’ (pipe) symbol e.g. AddrCode=(+WF.MF1+)|OfficerType=(+WF.OType+)

 

[LinkChanges=] Tag

By default, changes performed in the embedded screen will be linked automatically and appear as items in the workflow step along with selected linked items such as forms.

In the workflow, clicking on the icon will open the Form Queue screen with the corresponding form highlighted.

 

If the items are not required or you would like the item to be shown under certain condition, add the tag with the appropriate value:

  • [LinkChanges=N] will hide the change link in the workflow step

  • [LinkChanges=F] will show the change link only if linked forms have been selected and queued. If no forms were selected and queued, the change link will not be seen in the workflow step

  • [LinkChanges=L] will show the change only but hide the '+' sign for the change link item if no linked forms were selected and queued.

Note: These tags are also applicable for checklist item where field type is ‘Macros’ to show or hide the relevant link changes.

 

[LinkHolders=N] Tag

If documents are generated and stored within the workflow, the resulting document holder will be automatically linked to the workflow and added as an item in the workflow step.If this is not required, you can add the [LinkHolders=N] tag.

Configuring Item as Link to Time Entry/Disbursement Entry Screen

This allows user to create new time/disbursement entry without exiting the Workflow screen.

To do this, you only need to set the field type to ‘Time Entry’ or ‘Disbursement Entry’. However, you can set certain default values for the time/disbursement entry such as Service, Number of Units, Description and Matter in the Values/Default column.

If the workflow has a linked activity, the activity number of the workflow will be passed into the time/disbursement entry. This is only applicable when the All Time/Disbursement Entries Link to an Activity option is ticked. Refer to System Defaults - Time and Disbursements.

 

Specify Billing File

The Billing File for the time/disbursement entry will be the Master File of the workflow. However, it is also possible to have the Billing File be a Master File that is selected within the workflow. This requires an item with an assigned key and set as ‘Master File’ field type. This key will then be defined in the time entry details.

 

Specify 'Description' of entry

The description of the Time Entry/Disbursement Entry can be specified in Values/Default. You will need to have an item with an assigned key and configure the checklist item as ‘Text’ field type.

 

It is possible to have the Description retrieved from:

  • Items in the same workflow step using the (+<Key>+) tag.

  • Items in previous workflow step using the (+WF.<Key>+) tag.

Specify 'Matter' of entry

The Matter of the Time Entry/Disbursement Entry can be specified in Values/Default. You will need to have an item with an assigned key.

 

Configuring Item to Initiate or Link to Another Workflow

This allows user to start a new workflow (linked workflow) without exiting the current workflow (main workflow).

To do this, you only need to set the field type to ‘Workflow Link’ and specify details for the new workflow such as:

  • The key for the new workflow.

  • The Master File the new workflow is for.

  • Values for the new workflow from the current workflow.

To enter the details, click on the Browse button in Values/Default and add the necessary tags in the Note Editor.

In the workflow step, when there is a workflow link item, the following options will be seen:

  • Initiate new workflow - This is to start a new workflow from the current workflow.

  • Link to existing - This is to link the current workflow to a workflow that has been started.

 

Initiate New Workflow

To initiate a new workflow, click to select the step item and click Initiate new workflow.

Once the workflow is linked, the workflow icon will appear in the grid.

 

Link to Existing Workflow

When link to an existing workflow, the Select Workflow screen will be displayed. Depending on the workflow that you require, select the workflow status from Status drop-down list and apply the necessary filter.

You can also restrict the Status filter by adding the [WF.LinkStatus=] tag. For more information, refer to Checklist Item - Field Type: Workflow Link - Tags.

Once the workflow is linked, the workflow icon will appear in the grid.

 

Removing Workflow Link

If a workflow was incorrectly linked, this link can be removed by selecting the step item and clicking the Remove Link button.

When prompted, click Yes to confirm the removal. The workflow step will be blank so that another workflow can be linked instead.

 

Linking Approval Card to Workflow Step

To do this, ensure that the approval card has been set up first. Refer to Approval Cards.

Then set the field type to ‘Approval Card’ and in Values/Default, select the relevant Approval Card. If the workflow step contains more than one Approval Card, the required approvals for the first Approval Card must be completed before the approver(s) of the second Approval Card will be able to approve.

Configuring Item to Run File Review

This allows for File Review to be run from within the workflow. The item must be configured as 'File Review' field type and the scope of the review specified using the [Selection=] tag in Values/Default column with the relevant value - refer to Checklist Item - Tags. A change link will be appended under the item after the file review is run and the record will be linked to the workflow.

 

Configuring Item to Enable Duplication

In certain workflow steps, it may be required to have multiple instances of an item or a group of items. In this case, the option ‘Duplicate’ can be ticked to allow users to copy the item(s) in the workflow.

For a single item that is to be duplicated, you can click the Browse button in Values/Default column and tick the Duplicate option.

To allow a group of items to be duplicated, ensure that the items are grouped by:

  • Indenting the sub-items.

  • Ticking the Duplicate option for the first item which is now the parent item.

If you have previously ticked the Duplicate option for any of the sub-item, ensure that you untick it.

Once ticked, [Duplicate=1] is seen in Values/Default column.

Example – Step to get information for proposed new structure where user can enter multiple proposed names and statutory officers.

The configuration for the step requires:

  • Item to get the proposed name to have Duplicate option ticked.

  • Items related to statutory officers to be grouped with Duplicate option ticked for the parent item.

In the workflow step, user can click on the + icon to get another instance of the item or group:

When the Duplication option is ticked multiple times in a grouped item, a warning message will be prompted to prevent the workflow from being updated.

You are required to untick the option from the specified item.

Additionally, grouped items can be retrieved from the previous workflow step by using the [COPY.GROUP:<StepKey>.<HeaderKey>] tag. See Checklist Item - Tags.

 

Configuring Maximum Duplication

Duplications can be limited to a specific maximum number when needed. Click on the Browse button in Values/Default column in the Workflow Step Editor screen and specify the maximum number of duplicates that is appropriate for the item in the Duplicate Maximum field.

Ensure that the maximum number is set to more than 1 to allow duplication. The maximum number is inclusive of the original item(s). If the maximum number is set as 2, the item can only be duplicated once. As a result, there will be an original item and a duplicated item.

Configuration will be invalid if the following is being input:

  • Non-numeric(E.g.: ABC, One)

  • Less than or equal to 1

Invalid configuration will lead to unlimited duplication being allowed.

Configuring Item as Output for Automatic Decision

If the workflow contains decision steps, value of a checklist item can be used to automate the decision step. The checklist item is to be defined as an output for the decision.

 

Example – Workflow to perform file closure according to the selected Master File Type.

  • The decision step must have the input details for the Yes branch in terms of the Input Type and Value for the Yes branch

Configuring Item to Define Step Owner for Next Workflow Step

The owner of a workflow step within the same workflow can be defined using the 'System User' field type. The configuration is as follows:

  • The current workflow step must have the 'System User' field type set with a Key.

  • The 'Assign To' field in the 'Edit Step Properties' screen of another workflow step must be configured with the (-KEY.<ItemKey>-) code where <ItemKey> refers to the Key of the checklist item with the 'System User' field type.

If the key does not match, the workflow step will automatically be assigned to the workflow owner.

 

Example – To define the step owner for the subsequent workflow step for confirmation.

In the initial workflow step, set up a checklist item with the ‘System User’ field type and with a key.

For the workflow step where the owner is defined using the ‘System User’ field type, specify the ‘Assign To’ field with the (-KEY.User1-) code.

In the workflow, select a user to confirm the appointment.

 

Configuring Item to Create Invoices

This allows for an invoice to be created by the workflow in Viewpoint Connect and Engage with details from checklist items with the following field types:

  • Service

  • Lookup Value

  • User Table Value

  • Yes/No

Invoices created by workflow will be linked to the workflow and seen in the ‘Other Records’ tab. Depending on configuration, the invoice can either be draft or posted, and users will need to go to the corresponding screens to view the invoice. 

In the workflow step properties, the tag [ON_SUBMIT=CREATEINV] must be added as well. For all other field types EXCEPT ‘Service’ an additional tag – [CREATEINV.SERVCODE=<ItemKey>] is also required where <ItemKey> refers to the checklist item that will have the service(s) to be invoiced.

To define details for the invoice header, additional CREATEINV tag can be added. Refer to Checklist/Workflow Step - Tags.

Note: Using workflow to create invoice is only applicable for Master Files that have been set up as Billing Files with Matter(s). The services to be invoice must also exist in the Service Master.

 

Invoice Creation using ‘Service’ Field Type

This field type will create an invoice containing the service selected in the checklist item. The configuration requires the checklist item to be of the ‘Service’ field type with the following specified in Values/Default column:

  • Service Code – this can contain a single service code that will be invoiced, or multiple service codes for the user to select for invoicing.

  • Quantity Key – this is to contain the key of a checklist item that will contain the value for the number of units.

For the checklist item specified in Quantity Key, the field type can be either ‘Number’ or ‘Amount’ but the value entered must be a round number bigger than zero.

Note: If there is nothing set for Quantity Key, the system will look for a checklist item in the workflow step where the field type is ‘Number’ and it appears after the ‘Service’ item. The value entered for this checklist item will be the number of units for the service to be invoiced. If this is not available, then the number of units defaults to one.

 

Invoice Creation using ‘Lookup Value’ Field Type

For Lookup Value field type, this allows for creation of an invoice containing services that have been predefined in the lookup table value selected by the user. The workflow step requires a checklist item of the ‘Lookup Value’ field type with a key assigned and the lookup table specified through Values/Default.

In the Lookup Table, the service(s) is to be defined using the [SERVCODES=] tag.

 

Invoice Creation using ‘User Table Value’ Field Type

This is the same as Lookup Value field type except that the user is now selecting a user table value instead of a Lookup Table value. The service(s) to be invoiced must be defined using [SERVCODES=] tag for the user table value.

 

Invoice Creation using ‘Yes/No’ Field Type

This is used if the invoice is to be created when a user selects a Yes value. The checklist item must be set as ‘Yes/No’ field type and assigned a Key with [SERVCODES=] tag used to specify the service(s) to be invoiced in the Tag Values field for the checklist item.

Configuring Item Value as Running Number

This allows a checklist item’s value to be automated instead of requiring user input. For example, instead of having to enter the value ‘1’ for the first time, and then ‘2’ the next time, the item can be configured so that the value will be a running number. This requires the following configuration:

  • Set up a new lookup value in the Document Number Codes [DOCT] lookup table.

  • Add the new lookup value to System Running Numbers.

  • Specify the lookup value in the Values/Default field for the checklist item as (+SYS:DOCNUM.<Code>+) with the <Code> being the code of the new lookup table value.

 

Example – To populate running number for Matter Extension in workflow

In Configuration > Business Parameters > General > Lookup Values, add a new value to the Document Number Codes [DOCT] lookup table e.g. Matter Sequence Number [MTR].

In Configuration > Settings > Customise > System Running Numbers, add the new lookup value 'MTR' as a new document type. If you would like to include a prefix and/or set a format for the running number, specify this in the Prefix & Format field.

For the workflow checklist item, ensure to add the (+SYS:DOCNUM.<Code>+) code in the Values/Default.

In workflow, the running number for Extension will be populated automatically using the defined format.

 

Configuring Step or Item to Retrieve Rating Group

This allows for a workflow to utilise scores from different Rating Groups where the Rating Group can apply to the workflow step, or specific checklist items. The pre-requisites include:

  • Enabling the ‘Risk Rating’ option for the lookup value table or user table.

  • Assigning scores to the lookup table or user table values in the Risk Ratings screen for the correct Rating Group and Type.

Note: Ensure the correct Rating Group and Type are selected.

 

Once the risk rating has been set up for the lookup value table or user table, you can now include the [SCORETAG=<Code>.<Type>] tag in the Tag Values field of:

  • ‘Edit Step Properties’ screen

  • Items configured with Lookup Value or User Table Value field types

 

Configuring Tag in ‘Edit Step Properties’

If the workflow step requires only one rating group, add the tag to the workflow step’s properties in ‘Edit Step Properties’ screen.

 

Configuring Tag in Item with ‘Lookup Value’ or ‘User Table Value’ Field Type

When there are multiple checklist items within a workflow step that are included for risk rating, you can insert the tag in the Tag Values field and specify the Field/Mask field with the lookup table or user table where the risk rating is configured.

Note: If the same tag is applied for the field type as well as in Edit Step Properties screen, the program will take precedence over the field type.
