General System Settings

The available settings here allow for:

  • configuring version alert message

    The version alert message pops up when your databaseversion differs from the version of the program's executable files. The version number comprises of the following:

    • Major - This is the first part of the version number e.g. for the version number 8.0.15.200 this would be 8.

    • Minor - This is the second part of the version number e.g. for the version number 8.0.15.200, this would be 0.

    • Build- This is the third part of the version number e.g. for the version number 8.0.15.200, this would be 15.

    • Revision - This is the last part of the version number e.g. for the version number 8.0.15.200, this would be 200.

    You can specify when to show the message by entering the relevant value for the VERCHK Key:

    Value

    Result

    0

    Prompt warning message will be displayed when Major and Minor parts are the same but the Build part is different e.g. 8.0.15 vs 8.0.20.

    After the message is prompted, user may choose to continue/discontinue with the login process.

    2

    No prompt message, user may continue as normal.

    3

    No warning message. The user may continue as normal if the difference between the executable and database version is within the Version.AllowMinRevision and Version.Revision number in the Version table in the database.

    For instance, when the Version.AllowMinRevision number is 20 and the Version.Revision number is 15, then no warning will be displayed if the executable version is between 8.0.15 and 8.0.20.

    If the executable is not within this range, then a warning message will be displayed as per setting 0.

    Note: Version.Revision number in the database is actually referring to Build part.

    4

    Compatible Rule – Prompt error message will be displayed when Major and Minor parts are the same but the Build part is different. E.g.  8.0.20 vs 8.0.25

    This has the same behaviour as value 0. However, this will prompt an error message instead of a warning message. After the error message is prompted, the login process will discontinue and the application will shut down.

    Value 4 is the default system setting.

    5

    Strict Rule - Prompt message will be displayed when version number part is different. E.g. for version number 8.0.25.62, all 8.0.25.62 must be the same.

  • specifying Master File Details Report templates

    Master File Details reports list out information stored in Viewpoint in compliance with EU GDPR's Right of Access. There is a total of six reports and they are generated through Reporting > Word Reports. The design of the reports is based on the value of the MFDETR Key. To amend the report, you would amend the value.

    Key Value
    MFDETR The value is made up of tags that is divided into the following sections:

    • Templates - this is to specify the following:

      • [/FormCode=/] - this refers to the Form Code field.

      • [/Title=/] - this refers to the Description field and can be changed if required.

      • [/Content=/] - this refers to the information to be included such as C for Child Relations, P for Parent Relations and D for Documents.

      • [/Type=/] - this refers to the format which can be System Format (S) or Friendly Format (F).

      • [/HelpText=/] - this refers to the Help Text for the report and can be changed if required.

      • [/Notes=/] - this refers to the Notes field for the report and can be changed if required.

    • Properties - this is to specify the following:

      • [/CountryCode=/] - this refers to the Report Folder.

      • [/FormCat=/] - this refers to the Document Library for the report which is 1 for Word Reports.

      • [/LangCode=/] - this refers to the Language field.

      • [/HeaderSQL=/] - this refers to the header SQL for the report and can be changed if required.

      • [/SelectDef=/] - this refers to the Select Definition field.

    If you have made changes, you need to export the templates and import them. To export the templates, click Master File Details Report Templates in  Database Archiving > Troubleshooting.

  • configuring number of e-mails for TSNA services to process

    You can configure the number of e-mails to process per installation of Task Scheduler & Notification Agent services in Viewpoint Management. This can improve the handling of bulk e-mail processing if the number of e-mail notifications for a job causes performance issues.

    To set the number of e-mails for a single TSNA installation to process,

    1. From Viewpoint – Home, go to Configuration > Settings > Customise > System Settings.

    2. Select VPCROSS – TSNANR.

    3. Click Edit.

    4. Change the value in the Value field. Note:The default and recommended value for this is 100.

    5. Click Update.

    Each TSNA service installed will process the number of e-mails based on the value set for TSNANR. To improve bulk e-mail processing, it is encouraged to install more TSNA service. All installations will process the e-mails at the same time and with the sequence divided between the installations.

    For example, if the bulk e-mails number at 1000, TSNANR is set to 100, and there are two TSNA services installation:

    • TSNA 1 – processes 100 e-mail (1-100)

    • TSNA 2 – processes 100 e-mails (101-201)

    • TSNA 1 – processes 100 e-mails (202-302)

    • And the process will continue until all 1000 e-mails are processed.

  • configuring limit for e-mails sent for TSNA services

    You can configure the e-mail sending limit per installation of Task Scheduler & Notification Agent services in Viewpoint Management. This allows users to ensure that e-mails sent out do not exceed the limits set by e-mail service providers.

    The limit set is based on a 24-hour period. E-mails that exceed the limit will be processed in the next 24-hour cycle. To set the e-mail sending limit,

    1. From Viewpoint – Home, go to Configuration > Settings > Customise > System Settings.

    2. Select VPCROSS – TSNADL.

    3. Click Edit.

    4. Change the value in the Value field. Note:The default value is 0, indicating unlimited sending of e-mails.

    5. Click Update.

    Each TSNA service installed will process the number of e-mails based on the value set for TSNANR. For bulk e-mail sending, the limit defined by TSNADL will work concurrently with the value set for TSNANR.

    For example, with 1000 e-mails to send, TSNANR set to 100, TSNADL set to 500, and one TSNA service installation:

    • TSNA - Processes 100 e-mails

    • TSNA - Processes 100 e-mails

    • TSNA - Processes 100 e-mails

    • TSNA - Processes 100 e-mails

    • TSNA - Processes 100 e-mails

    The TSNA will process up to 500 out of the 1000 e-mails within 24 hours. The remaining 500 e-mails will be processed in the next 24-hour cycle.

    Note: E-mails processed will never exceed the limit defined by TSNADL regardless of any remaining process slots from the TSNANR value. 

     

Note: These settings are in the VPCROSS application.

  • customising ESP display information

    The ESP Entities list allows customisation of the Entity's display information and Master File compliance status or any defined status in colour badge. This can be done by configuring tags for both ENTADD and ENTLST Keys. The task scheduler job 'Engage Servicing Portal - Entity Update [CPEU]' retrieves Back-office records based on the queries stored in System Settings ‘ENTADD’ key and stores the records in the Front-office database. The records will then be displayed in the ESP 'Entities' list.

    The ENTADD Key stores SQL queries that retrieves data from the Back-office database.

    Key

    Value

    ENTADD

    Syntax:

    [/EntityField.sql=SELECT <column_name> FROM <table_name> WHERE <entity_column_name>=‘(-MF-)’/]

     

    Default Value:

    [/EntityField.sql=/]

     

    Example Tag:

    [/EntityField.sql= SELECT AdminGroup FROM RefMaster WHERE RefCode=‘(-MF-)’/]

     

    Multiple queries can be entered by separating each with a ‘;’. See below syntax:

    [/EntityField.sql= SELECT <column_name> FROM <table_name> WHERE <entity_column_name>=‘(-MF-)’;

    SELECT <column_name> FROM <table_name> WHERE <entity_column_name>=‘(-MF-)’/]

     

    Example tag:

    [/EntityField.sql= SELECT AdminGroup FROM RefMaster WHERE RefCode=‘(-MF-)’;

    SELECT EntClient FROM Entity WHERE EntCode=‘(-MF-)’;

    SELECT AddrCode from Officers WHERE EntCode=‘(-MF-)’;

    SELECT KYC_Status FROM RefMaster WHERE RefCode =‘(-MF-)’/]

    The ENTLST key stores the display settings of the Entity’s customised fields and Master File status colour badge.

    Key

    Value

    ENTLST

    Tag syntax for the Entity’s customised fields:

    Syntax:

    [/Ent.cols=<column_name>,<column_name>,<column_name>/]

     

    Default Value:

    [/Ent.cols=JurType, EntType/]

     

    Example Tag:

    [/Ent.cols=AdminGroup, EntType, JurType/]

     

    Tag syntax for Master File status ‘colour badge’:

    Syntax:

    [/Ent.badge=<badge ID>/]

    [/Ent.badge.<badge ID>.title=<badge title>/]

    [/Ent.badge.<badge ID>.colour=<any CSS colour name or colour code>/]

    [/Ent.badge.<badge ID>.condition=<status condition defined in SQL query>/]

    Note: You can define more than one badge ID in the tag [/Ent.badge=<badge ID>/] by separating each with a comma (,).

     

    Example Tag:

    [/Ent.badge=comp,noncomp/]

    [/Ent.badge.comp.title=Compliance/]

    [/Ent.badge.comp.colour=Green/]

    [/Ent.badge.comp.condition=KYC_Status=09/]

    [/Ent.badge.noncomp.title=Non-Compliance/]

    [/Ent.badge.noncomp.colour=Red/]

    [/Ent.badge.noncomp.condition=KYC_Status<>09 OR KYC_Status IS NULL/]

Practice Management System Settings

The available settings here allow for:

  • using invoice line current amount to calculate base VAT amount

    You may opt to calculate Base VAT amount using Invoice line currency amount in the Practice Management.

    Key

    Value

    INVOPT

    1

    When INVOPT = 1, Base VAT amount will be calculated using Invoice line currency amount (LineInvCur).

    Conversion to Base VAT amount = Invoice line currency amount * VAT % * exchange rate.

    Note: 'Invoice line currency amount' is referring to the value stored in 'LineInvCur' field.

    When INVOPT is not defined, the Base VAT amount will be calculated using Invoice line currency (VatLineCur).

    Conversion to Base VAT amount = VAT amount (rounded to 2 d.p) in invoice line currency * exchange rate.

    Note: 'VAT amount (rounded to 2 d.p) in invoice line currency' is referring to the value stored in 'VatLineCur' field.

     

  • defining the entity status and entity sub-status for strike-off

    For Fees Payable - Strike Off, the program will check entities that are due to be struck off based on the Strike Date for the entity and update the entity status and entity sub-status accordingly to the value set for the following Keys:

    Key

    Description
    STKOFF This is to specify the new entity status when the entity has been struck off. The value is the code for the entity sub-status maintained in the Entity Sub-Status [COSU] lookup table.
    STKSUB This is to specify the new entity sub-status when the entity has been struck off. The value is the code for the entity sub-status maintained in the Entity Sub-Status [COSU] lookup table.
