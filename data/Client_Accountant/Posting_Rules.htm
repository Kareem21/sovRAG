Posting Rules

This feature is used when you want to post journals based on a set of rules. There are four different types of rules that can be created:

  • Appropriate Deferred Income

    This rule type is used for deferred income/expense, and will calculate and divide a fixed amount from a designated "source" account for periodic release into a "release" account.

    Deferred income is income that is received for goods or services that have yet to be supplied or rendered. An example of this would be when you bill a client in advance for the services to be provided for the coming year, such as for management fees. The full amount cannot yet be recognised as income and is thus placed in a liability account, to be progressively moved to an income account as the year progresses.

    An item that gives rise to deferred income is prepayment, or deferred expense. If you are keeping client accounts, then the payment made by the client to you for services-to-be-rendered, is considered prepayment. In the client's accounts, the amount is placed in an asset account and progressively moved to an expense account as the year progresses. If you also pay a supplier in advance for goods or services to be supplied or rendered, then this would also be recorded as a deferred expense.

    In the explanation given above, the amounts posted to the source account originate from the Billing module. In this case, the following must be done in order to use this rule type:

    • Account mapping - Account mapping must be done for the services that feed the source account.

    • Posting rule - This must be set up correctly to ensure that each source account feeds the correct release account.

    • Exchange rates - All relevant exchange rates must be set up.

    • Beginning and end dates - The beginning and end dates for the post to the source account determine how long the rule will run. These dates are represented by the Date - From and Date - To fields of the detail line of the invoice. The same fields must be completed for supplier invoices. The Date - From will be updated as Line Date whereas the Date - To will be updated as Reconcile Date in the invoice's journal.

    You may also create a journal to feed the source account directly in Accounting. In this case, you need to enter the beginning and end dates of the period into the fields that are represented by the Line Date and Reconciliation Date fields in the Journal screen.

    When the rule is run, the system calculates the number of days in the specified run period and divides that by the number of days specified by the beginning and end dates in the invoice or journal. This number is then multiplied by the amount of the invoice or journal. This final amount is moved from the source to the release account.

    For example:

    You have an invoice for Director fees for the period from 1 January to 31 December amounting to EUR12,000.00.

    If you run this rule from 1 January to 31 March, the amount moved from the source account to the release account would be calculated as follows:

    (31+28+31)

    x  EUR12,000.00

     = EUR2,958.90

    365

  • Debtors Provision Accrual

    This feature posts bad debt provisions from invoices created in Billing, and reverses them the next day. It is a useful management reporting tool, and is meant for use by Revenue Entities only. When run, the system checks through all outstanding invoices for bad debt provisions and compares them to the invoice amount based on the date of the run.

    • If the provision amount is lower than the invoice amount, the provision amount is posted.

    • If the provision amount is higher than the invoice amount, the invoice amount is posted.

    If you utilise the organisational units feature, then the provisions are grouped by organisational unit.

    The following must be done In order to use this rule type:

    • Account mapping - This is done in System Manager, and must be done correctly for the components specified below; which is found under Control Accounts - Sales Ledger for the selected Revenue Entity. If the components cannot be seen, then activate them in the Account Types Practice Management [PMAT] lookup table.

      • Debtors Provision (Credit) [R.SO] - This is the nominal account where provisions are credited to.

      • Debtors Provision (Debit) [R.SP] - This is the nominal account where provisions are debited to.

    • Posting rule - This must be set up correctly so that the appropriate journal reference and line description is used for the journal entries.

    Debtor provision calculation can be set in the tag value of Posting Rule:

    [ProvisionXX_Y=t]

    [ProvisionXX_Percent=P]

    [Provision_Method=X]

    [Provision_DateType=Z]

    Where:

    • XX is a sequence number in 2 digit format, 01, 02, 03 to 10 with maximum ten brackets.

    • Y is the period from the invoice due date. Y denotes Year, M denotes Month, W denotes Week and D denotes Day.

    • P is percentage of provision, valid range from 1 to 100.

    • X can be A for Ageing or I for Invoicing.

    • Z can be DateInvoice or DateDue.

    Example 1

    [Provision01_Y=2][Provision01_Percent=100]

    [Provision02_M=18][Provision02_Percent=80]

     

    This would mean that provision amount for Invoice due for two years and above will be 100% of the outstanding amount. Meanwhile, Provision amount for Invoice due for 18 months and above will be 80% of the outstanding amount.

     

    Example 2

    [Provision01_Y=2][Provision01_Percent=100]

    [Provision02_M=18][Provision02_Percent=80]

    [Provision03_M=12][Provision03_Percent=60]

    [Provision04_d=120][Provision04_Percent=40]

    [Provision05_d=45][Provision05_Percent=30]

     

    An Invoice amounting EUR1000 is created on 1 Jan 2016. A receipt for this invoice is issued on 1 August 2016 with the amount of EUR100.

    When the user runs the Debtor provision rule on 31 July 2016 with Tag Values defined above, it identifies that it has exceeded 120 days but yet to reach 12 months. This provision rule will apply:

    [Provision01_d=120][Provision04_Percent=40]

    The provision amount will be 40% of the unsettled debt.

    Hence on 31 July 2016, Provision amount = EUR1000 X 40% = EUR400.

    However, when the user runs the Debtor provision rule on 1 August 2016 (or onwards), the debtor has settled an amount of EUR100, resulting in the total of unsettled debt to EUR1000-EUR100 = EUR900. Hence the Provision amount = EUR900 X 40% = EUR360.

    The provision amount will be posted to the journal.

    When the rule is run, the sum of all bad debts up to the Reconcile date is credited to the Debtors Provision (Credit) [R.SO] account. This credit is matched by a debit to the Debtors Provision (Debit) [R.SP] account. Both transactions use the Reconcile date as the journal date. The entries are then immediately reversed, using the next day's date as the journal date.

     

    By default, Debtors Provision Accrual calculates the amount based on the invoice 'Due Date'.

    It also supports calculation based on 'Invoice Date' by adding a tag line [Provision_DateType=DateInvoice].

    Example 3

    [Provision03_M=12][Provision03_Percent=60]

    [Provision04_D=120][Provision04_Percent=50]

    [Provision05_D=45][Provision05_Percent=30]

    [Provision_DateType=DateInvoice]

    By applying the tag [Provision_DateType=DateInvoice], the amount calculation will be based on the Invoice Date, hence the outcomes are:

    • Provision amount for Invoice due for 2 years and above will be 100% of the outstanding amount. In this example, provision amount = 0.

    • Provision amount for Invoice due more than 18th months and less than 2 years will be 80% of the outstanding amount. In this example, provision amount = 2400 (from Invoice 1 and 2).

    • Provision amount for Invoice due more than 12th months and less than 18th months will be 60% of the outstanding amount. In this example, provision amount = 0.

    • Provision amount for Invoice due more than 120 days and less than 12 months will be 50% of the outstanding amount. In this example, provision amount = 3500 (from Invoice 3 and 4).

    • Provision amount for Invoice due more than 45 days and less than 120 days will be 30% of the outstanding amount. In this example, provision amount = 1500 (from Invoice 5).

    Therefore, the total amount is 2400 + 3500 + 1500 = 7400.

     

    If the tag [Provision_DateType=DateInvoice] is excluded, the amount calculation will be based on the respective invoice’s Due Date, hence the outcomes are:

    • Provision amount for Invoice due for 2 years and above will be 100% of the outstanding amount. In this example, provision amount = 0.

    • Provision amount for Invoice due more than 18th months and less than 2 years will be 80% of the outstanding amount. In this example, provision amount = 2400 (from Invoice 1 and 2).

    • Provision amount for Invoice due more than 12th months and less than 18th months will be 60% of the outstanding amount. In this example, provision amount = 0.

    • Provision amount for Invoice due more than 120 days and less than 12 months will be 50% of the outstanding amount. In this example, provision amount = 1500 (from Invoice 3).

    • Provision amount for Invoice due more than 45 days and less than 120 days will be 30% of the outstanding amount. In this example, provision amount = 2700 (from Invoice 4 and 5).

    Therefore, the total amount is 2400 + 1500 + 2700 = 6600.

     

    Provision Method Invoicing [Provision_Method=I]

    When using Provision Method I, the provision amount will be calculated as follows:

    • When a 'Provision Amount > 0' is specified in the Settlement > Notes screen and the invoice outstanding amount is > 0, the provision is calculated as the smallest of the 'Provision Amount' and the 'Outstanding amount'.

    Example 1

    [Provision_Method=I]

    This means that the system will include all invoices with a Due Date <= Rule run date.

     

    Example 2

    [Provision_Method=I]

    [Provision_DateType=DateInvoice]

    This means that the system will include all invoices with Invoice Date <= Rule run date.

     

    Note: If no [Provision DateType] is specified, the system will check the 'Due Date' by default.

     

    The system will apply the Ageing method if the below condition is met:

    • Only [Provision01] tag is being defined or;

    • [Provision_Method=A] and [Provision01] tags are defined

    The system will apply the Invoicing method if the below condition is met:

    • No tag is defined or;

    • [Provision_Method=A] tag is defined without [Provision01] tag or;

    • [Provision_Method=I] tag is defined

    The system will apply both the Ageing and Invoicing method if the below condition is met:

    • [Provision_Method=I] and [Provision01] tags are defined

    For Invoices with a Provision amount, the Invoicing method will be used, and for Invoices without Provision amount, the Ageing method will be used.

     

    Where do I specify the bad debt amount?

    This is done by selecting the invoice and clicking Notes in Sales Ledger > Accounts Receivable > Settlements.

     

  • Recurring Fees in Arrears Accrual

    This feature posts recurring fee lines that are:

    • In arrears - The selection for the In field in the Recurring Fee set up in Billing is Arrears [R].

    • Active - Only active fees are picked up.

    • Not yet pulled into a draft invoice - This means that the Draft No. is 0.

    • Due after the reconcile date - This means that the Next Due Date is after the Reconcile Date selected for the posting rule run.

    • Effective before the reconcile date - This means that the fee is effective before the Reconcile Date selected for the posting rule run.

    • Not in the Minimum Time Fee [RM] service class - The time entered for this service class will already be picked up by the WIP Accrual Posting rule.

    The following must be done to use this rule type:

    • Account mapping - This needs to be done correctly for the following components under the Control Accounts - Sales Ledger for the selected Revenue Entity:

    • WIP - Fees in Arrears - This is the nominal account where fees are debited to on the Reconcile Date, and credited to on the next day.

    • WIP - Arrears offset - This is the nominal account where time fees are credited to on the Reconcile Date and debited to on the next day.

    • Posting rule - This must be set up correctly, so that the appropriate journal reference and line description are used for the journal entries.

    You are allowed to specify account used for posting and also the scope of the rule. This can be configured under Noted/Tag Values.

    Tag configuration:

    [R.SF=X]

    [R.SG=Y]

    [R.SERVCODE=A,B,C]

    Where:

    • X is posting account code for 'WIP - Fees in Arrears'.

    • Y is posting account code for 'WIP - Arrears offset'.

    • A,B,C is the list of service codes to be included in the scope.

    These will override setting in account mapping R.SF and R.SG.

    Note:

    • Only Account type 1-4 for X and Y is allowed.

    • R.SF and R.SG must set in pair.

  • WIP Accrual Posting

    This feature posts accrual income from works-in-progress (WIP) in the Time and Disbursements module, and reverses them the next day, which may be useful for management reporting purposes. When run, the system checks through all approved time and disbursement entries and ignores unapproved ones. As with the Debtors Provision Accrual rule type, if you are utilising the organisational units feature, then the WIP amounts for time and disbursements are grouped by organisational unit. This rule is meant for use on Revenue Entities only.

    The following must be done in order to use this rule type:

    • Account mapping - This needs to be done correctly for the following components under the Control Accounts - Sales Ledger for the selected Revenue Entity:

      • WIP - Time charges [R.SX] - This is the nominal account where time charges are debited to.

      • WIP - Time offset [R.ST] - This is the nominal account where time charges are credited to.

      • WIP - Disb. charges [R.SY] - This is the nominal account where disbursement charges are debited to.

      • WIP - Disb. offset [R.SU] - This is the nominal account where disbursement charges are credited to.

    • Posting rule - This must be set up correctly, so that the appropriate journal reference and line description are used for the journal entries.

    When the rule is run, the sum amount of all approved WIP are debited and credited to the respective accounts. Immediately after, this accrual journal is reversed, using the next day's date as the journal date.

    Viewpoint picks up all approved WIP as at the date entered. As this date can be in the past, the sum amount will include any works that may have since been invoiced.

     

    The following tags can be used in Note/Tag Values field.

    Tag

    Description

    [TDTYPE.EXCL=DISB]

    This tag excludes Disbursement in WIP Accrual posting rule.

    [TDTYPE.EXCL=TIME]

    This tag excludes Time in WIP Accrual posting rule.

    [MTSTATUS.EXCL=<Status>]

    This tag excludes time and disbursement entries based on Matter File status. The tag supports multiple status, separated with a comma. For example: [MSTATUS.EXCL=2,5] will exclude Matter File Status 2 and 5.

    [BFSTATUS.EXCL=<Status>]

    This tag exclude time and disbursement entries based on Billing File status. The tag supports multiple status, separated with a comma. For example: [BFSTATUS.EXCL=2,5] will exclude Billing File Status 2 and 5.

    [GROUP_XX=<Days>][PERCENT_XX=<Percentage>]

    This tag will determine which of the approved Time and Disbursement entries as well as the percentage applied that will be calculated in WIP Accrual posting rule. Note that XX can only support from 01-10, while <Days> and <Percentage> can only support whole numbers. For more information, refer to examples below.

Command buttons

These apply to the posting rule list. Click the relevant command button to perform the necessary.

  • New - Click this to add a new rule.

    1. Enter the information required to set up your rule.

    2. Click Update.

  • Edit - Click this to edit a rule.

  • Delete - Click this to delete a rule.

  • Run rule - Click this to run a specific rule, or, if the Run-all checkbox has been ticked, all rules at once. The run rule date range needs to be within the 48 periods open of an Account File. If the rule is run for the first time, the date control is enabled where user can set the date range for the rule run. For the subsequent run, the date control will be semi-enabled. For Appropriate Deferred Income [DI] posting rule, the Reconcile option in the 'Run Rule' window allows user to reconcile right after the run.

    When the Reconcile option is ticked, the system will run rule from 01/01/1900 in addition to the 'From' date. Hence, you will see two rules being posted after the successful run.

    • Appropriated Deferred Income [DI] - the 'From' date will default to 'the last succeeded run date + 1 day' and is not editable (date control is disabled). The 'to' date is set to 'the last day of the particular calendar period' and is editable (date control is enabled). Note: There is a validation on the 'to' date even though it is editable. However, it is limited to not exceed the current period’s last day.

    • Other Posting Type -  the 'On Date' will default to the following month-end date based on the last succeeded run month and is editable (date control is enabled).

  • Reconcile - Click to reconcile up to the date of last succeeded rule run. This will calculate the Amount Left in the Posting Rule Run Details report. This function is used to clear up left-over balances when run at the end of your accounting period.

Note: Upon upgrading from a lower version, it is recommended to reconcile 'Appropriated Deferred Income [DI]' posting rule. Note that the Run Rule or Reconcile will take longer than usual for the first time to calculate/correct the balance and release amount for history run records.
